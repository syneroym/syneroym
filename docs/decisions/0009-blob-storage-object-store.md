# ADR-0009: Blob Storage Backend via `object_store` (D-03-04)

## Status

Accepted

## Context

`[PLT-DAT]` requires an object (blob) storage service for `SynSvcs`. Blobs are
content-addressed by SHA-256. The service must support local filesystem storage
for development and S3-compatible object storage for production, with a unified
API surface exposed to WASM guests via WIT host functions.

Two approaches were evaluated:

- **Custom local + S3 backends:** Write a thin `BlobService` trait with separate
  filesystem and S3 implementations. Avoids a new dependency at the cost of
  duplicated async I/O and HTTP signing code.
- **`object_store` crate (Apache Arrow):** A production-ready unified async Rust
  API supporting local filesystem, AWS S3 (and S3-compatible endpoints such as
  MinIO), GCS, and Azure Blob Storage behind a single `ObjectStore` trait.

## Decision

**Use the `object_store` crate from Day 1** as the unified blob backend.

### Rationale

- The `object_store` crate is pure async Rust with no C FFI. It is
  cross-platform (Linux, macOS, Windows) and widely deployed in production
  (Apache DataFusion, Delta Lake, InfluxDB).
- The dependencies it pulls in (`aws-sigv4`, `reqwest`, `hyper`, `rustls`) are
  already present in the workspace. Incremental compile cost is modest.
- The local filesystem backend in `object_store` (`LocalFileSystem`) is
  production-quality. There is no need to write or maintain a custom filesystem
  implementation.
- A single `Arc<dyn ObjectStore>` field in `BlobService` makes the backend
  switchable via configuration with zero code changes. Tests use
  `InMemory` or `LocalFileSystem`; production uses `AmazonS3` or compatible.

### Storage Path Layout

Blobs are stored under a per-service namespace:

```
<blobs_root>/<service_id>/<aa>/<remaining-62-hex-chars>
```

The two-character directory prefix (first two hex chars of the SHA-256 digest)
balances inode usage for filesystems with per-directory entry limits.

For `object_store` backends, the same path string is used as the object key,
preserving the layout convention across local and cloud backends.

### Content Integrity

- On `put-blob`, the host computes SHA-256 of the supplied bytes and uses it as
  the storage key (content-addressed). The returned key is the 64-character
  lowercase hex digest.
- On `get-blob`, the host re-computes SHA-256 of the retrieved bytes and rejects
  the blob with `blob-error::internal("integrity check failed")` if the digest
  does not match the requested hash. This detects silent storage corruption.

### Path Traversal Guard

- `service_id` must match `^[a-zA-Z0-9_\-]{1,128}$` before any path is formed.
- `hash` must be exactly 64 lowercase hex characters (`^[0-9a-f]{64}$`).
- The host validates both constraints with `Err(blob-error::internal(...))` on
  violation, without touching the filesystem or object store.
- `Path::join` + `starts_with` is used to verify the final resolved path is
  strictly a descendant of `<blobs_root>` (not `Path::canonicalize`, which
  requires the path to exist and behaves differently across OSes).

### Signed URLs

HMAC-SHA256 presigned URLs are generated by the host using a substrate-local
signing key stored in `substrate.db`. The URL format is:

```
GET /blobs/<hash>?exp=<unix-ts>&sig=<base64-url-hmac>
```

The substrate serves blobs via an HTTP endpoint that validates `exp` (TTL) and
`sig` before streaming the blob from the `ObjectStore`. The signing key is
rotated by the operator via `roymctl blob-signing-key rotate`.

Public HTTP serving (CDN integration, public access without a signature) is
**deferred** — all blob access in M3 requires a valid presigned URL.

### WIT Interface

```wit
interface blob-store {
    variant blob-error {
        not-found,
        quota-exceeded,
        internal(string),
    }

    put-blob:   func(data: list<u8>) -> result<string, blob-error>;
    get-blob:   func(hash: string)   -> result<list<u8>, blob-error>;
    delete-blob: func(hash: string)  -> result<_, blob-error>;
    signed-url: func(hash: string, ttl-secs: u32) -> result<string, blob-error>;
}
```

### Configuration

```toml
[storage.blobs]
backend = "local"               # or "s3"
local_root = "/var/lib/syneroym/blobs"

[storage.blobs.s3]              # populated when backend = "s3"
endpoint = "https://s3.amazonaws.com"
bucket   = "my-syneroym-blobs"
region   = "us-east-1"
# Credentials via environment variables (AWS_ACCESS_KEY_ID, etc.)
```

## Consequences

- `object_store` is added to `Cargo.toml` workspace dependencies.
- `crates/blob-store/` is a new crate wrapping `Arc<dyn ObjectStore>` with the
  content integrity verification, path guard, and presigned URL logic.
- The blob service is not WASM-compiled; it runs exclusively in the native
  substrate host binary.
- Tests use `object_store::memory::InMemory` to avoid filesystem I/O; integration
  tests use `object_store::local::LocalFileSystem` with a `tempdir`.
- S3-compatible production backends (MinIO, Tigris, Cloudflare R2) are supported
  by setting `endpoint` to the compatible endpoint URL.
