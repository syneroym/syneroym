> **Status:** planning (not started). Branch note: `feat/m04a-a1` already

## 0. Summary of the change

B0 shipped the *capability model* (`ResourceUri`, `Ability`, `Capability`,
`SessionContext`) and the interim admin-root path in `build_caller`
(`io.rs:47`), but deferred the actual **UCAN token chain** and its verification
(`crates/ucan/src/lib.rs:5-8` explicitly). B1 delivers exactly that deferred
half and wires it into ingress:

1. **`syneroym-ucan`**: add `CapabilityToken` (signed delegation token),
   `CapabilityToken::issue`, `verify_chain`, `SessionContext::from_verified_chain`,
   `Capability::covers`, and the `AuthNormalizer` trait + `DidKeyNormalizer`
   (ADR-0015 §5 seam, did:key only for M4).
2. **`syneroym-identity`**: add one reusable verify helper
   (`substrate::verify_json_signature`) so `syneroym-ucan` verifies signatures
   without pulling in `ed25519-dalek`/`z32` directly.
3. **`syneroym-router`**: carry a UCAN token on the preamble (`ucan=` param,
   mirroring `delegation=`), verify it in `build_caller`, and populate
   `CallerContext.session.capabilities` with `AuthLevel::Ucan`. Reuse the
   existing DHT-revocation resolver for chain-DID revocation.
4. **Tests**: exhaustive `verify_chain` unit tests (fail-closed on every
   escalation/tamper/audience-mismatch/expiry path) + an ingress integration
   test proving reference-scenario **step 21** (a client presents a UCAN → the
   node verifies the chain → a scoped capability reaches native dispatch).

**Trust model B1 commits to (read this first — it resolves the main ambiguity
in the task, see Flag F1):** the node's **admin root** (`admin_ucan_root`, or
the verified `ControllerAgreement` controller that overrides it at boot —
`runtime.rs:345`) is the *sole* trusted root issuer. Every capability in a
presented chain must attenuate back to a token issued by that root. Per-service
**owner**-rooted chains (owner ≠ node admin) are **not** verifiable at B1
because the app catalog records no owner DID yet — that is Slice **B7**. This
makes B1 a strict generalization of B0's direct-equality admin path: B0 granted
`substrate/admin` iff `caller_did == admin_root`; B1 additionally accepts a
*chain* rooted at `admin_root` that delegates narrower capabilities to the
caller.

## 1. `syneroym-identity`: reusable signature-verify helper

### 1.1 `crates/identity/src/substrate.rs`
`Identity::sign_json` (keys.rs:199) already canonicalizes a `Value` (RFC-8785)
and returns a z-base-32 signature. Add its inverse as a free function so the
`ucan` crate does not need `ed25519-dalek`/`z32` deps of its own:

```rust
/// Verify a z-base-32 Ed25519 signature (as produced by `Identity::sign_json`)
/// over the RFC-8785 canonicalization of `value`, against the pubkey resolved
/// from `signer_did`. Mirrors `DelegationCertificate::verify`'s crypto steps.
pub fn verify_json_signature(signer_did: &str, value: &Value, sig_z32: &str) -> Result<()> {
    let pubkey = resolve_did_key(signer_did)
        .context("failed to resolve signer DID")?;
    let canonical = canonicalize_json_value(value);
    let payload = serde_json::to_string(&canonical)
        .context("failed to serialize canonical payload")?;
    let sig_bytes = z32::decode(sig_z32.as_bytes())
        .map_err(|_| anyhow!("invalid z-base-32 signature encoding"))?;
    let signature = Signature::from_slice(&sig_bytes)
        .context("invalid Ed25519 signature bytes")?;
    pubkey.verify(payload.as_bytes(), &signature).map_err(Into::into)
}
```
`Signature`/`Verifier`/`VerifyingKey` are already imported at
`substrate.rs:8`. Unit test: sign a `Value` with `Identity::sign_json`, verify
with the new fn against the signer's `derive_did_key`; assert a tampered value
and a wrong signer both fail.

### 2.2 New module `crates/ucan/src/token.rs`

```rust
use anyhow::{Result, anyhow};
use serde::{Deserialize, Serialize};
use serde_json::{Map, Value};
use syneroym_identity::{Identity, substrate};

use crate::capability::Capability;

/// Clock-skew tolerance for `not_before` (mirrors `DelegationCertificate`'s
/// 300 s future-issue tolerance).
const CLOCK_SKEW_SECS: u64 = 300;

/// A signed UCAN capability token (ADR-0015 §2). `proofs` are the parent tokens
/// forming the delegation chain. The Ed25519 signature covers the RFC-8785
/// canonicalization of every field **except `signature` and `proofs`** — each
/// proof is independently signed by its own issuer, and chain continuity
/// (`proof.audience == child.issuer`) binds them, so a valid proof cannot be
/// repackaged under a child the proof's issuer never signed for.
#[derive(Clone, Debug, PartialEq, Eq, Serialize, Deserialize)]
pub struct CapabilityToken {
    pub issuer_did: String,
    pub audience_did: String,
    pub capabilities: Vec<Capability>,
    /// Proven claims surfaced into `SessionContext.claims` (co-design seam #1:
    /// M04B binds these as SQL `?` params). Empty by default.
    #[serde(default)]
    pub facts: Map<String, Value>,
    pub not_before_secs: u64,
    pub expires_at_secs: u64,
    #[serde(default)]
    pub proofs: Vec<CapabilityToken>,
    /// z-base-32 Ed25519 signature over the canonical body (sans this field
    /// and `proofs`).
    pub signature: String,
}
```

**Signed-body construction** (shared by `issue` and per-token verification):
```rust
impl CapabilityToken {
    fn signing_value(&self) -> Value {
        // Serialize self, then drop the two non-signed fields. Reusing the
        // serde form keeps field names in exact sync with the struct.
        let mut v = serde_json::to_value(self).expect("CapabilityToken is serializable");
        if let Some(obj) = v.as_object_mut() {
            obj.remove("signature");
            obj.remove("proofs");
        }
        v   // canonicalization happens inside sign_json / verify_json_signature
    }
}
```

**`issue`:**
```rust
pub fn issue(
    issuer: &Identity,
    audience_did: &str,
    capabilities: Vec<Capability>,
    facts: Map<String, Value>,
    expires_in_secs: u64,
    proofs: Vec<CapabilityToken>,
) -> Result<Self> {
    let issuer_did = substrate::derive_did_key(&issuer.public_key());
    let now = now_secs()?;                       // SystemTime → UNIX secs helper
    let mut token = Self {
        issuer_did,
        audience_did: audience_did.to_string(),
        capabilities,
        facts,
        not_before_secs: now,
        expires_at_secs: now + expires_in_secs,
        proofs,
        signature: String::new(),
    };
    token.signature = issuer.sign_json(&token.signing_value())?;
    Ok(token)
}
```

**Per-token structural verification** (signature + time bounds, no chain walk):
```rust
fn verify_self(&self, now_secs: u64) -> Result<()> {
    if self.not_before_secs >= self.expires_at_secs {
        return Err(anyhow!("token has non-positive validity window"));
    }
    if self.not_before_secs > now_secs + CLOCK_SKEW_SECS {
        return Err(anyhow!("token not_before is in the future"));
    }
    if now_secs >= self.expires_at_secs {
        return Err(anyhow!("token expired"));
    }
    substrate::verify_json_signature(&self.issuer_did, &self.signing_value(), &self.signature)
        .context("token signature verification failed")
}
```

### 2.3 Chain verification (`crates/ucan/src/token.rs` cont.)

Trust is injected as a predicate so the crate stays policy-agnostic and pure
(the router supplies "issuer == admin_root", see §4):

```rust
/// Options for verifying a presented token chain.
pub struct ChainVerifyOpts<'a> {
    /// The DID this token must be addressed to (the verified connection
    /// identity). The leaf's `audience_did` must equal this — binds the token
    /// to the presenter, preventing replay of a token issued to someone else.
    pub expected_audience_did: &'a str,
    /// Returns whether `issuer_did` is a trusted root of authority for
    /// `resource`. At B1 the router passes `|iss, _res| iss == admin_root`.
    pub is_trusted_root: &'a dyn Fn(&str, &crate::ResourceUri) -> bool,
    pub now_secs: u64,
}

/// Verify the chain and return the set of capabilities the leaf provably holds
/// — i.e. each is either rooted directly at a trusted issuer or attenuated from
/// a valid, trusted proof. Capabilities that cannot be traced to a trusted root
/// are **dropped** (fail-closed at the capability granularity); an entirely
/// unbacked leaf yields an empty set, not an error. Returns `Err` only on a
/// structural failure (bad signature, expiry, audience mismatch).
pub fn verify_chain(leaf: &CapabilityToken, opts: &ChainVerifyOpts) -> Result<Vec<Capability>> {
    if leaf.audience_did != opts.expected_audience_did {
        return Err(anyhow!(
            "token audience {} does not match presenter {}",
            leaf.audience_did, opts.expected_audience_did
        ));
    }
    granted_capabilities(leaf, opts)
}

fn granted_capabilities(token: &CapabilityToken, opts: &ChainVerifyOpts) -> Result<Vec<Capability>> {
    token.verify_self(opts.now_secs)?;         // fail-closed: bad link aborts
    let mut effective = Vec::new();
    // Verify proofs once; a proof that fails structurally makes the whole
    // presentation invalid (do not silently ignore a tampered proof).
    let parent_grants: Vec<Vec<Capability>> = token
        .proofs
        .iter()
        .map(|p| {
            if p.audience_did != token.issuer_did {
                // continuity break: this proof does not delegate to this issuer
                return Ok(Vec::new());
            }
            granted_capabilities(p, opts)
        })
        .collect::<Result<_>>()?;
    for cap in &token.capabilities {
        let rooted = (opts.is_trusted_root)(&token.issuer_did, &cap.with);
        let backed = parent_grants
            .iter()
            .flatten()
            .any(|pc| pc.covers(cap));
        if rooted || backed {
            effective.push(cap.clone());
        }
        // else: dropped (fail-closed) — issuer is neither a trusted root nor
        // holds a proof entailing this capability.
    }
    Ok(effective)
}
```

`SessionContext::from_verified_chain` (in `session.rs`) wraps it:
```rust
impl SessionContext {
    pub fn from_verified_chain(leaf: &CapabilityToken, opts: &ChainVerifyOpts) -> Result<Self> {
        let capabilities = crate::verify_chain(leaf, opts)?;
        Ok(Self {
            subject_did: leaf.audience_did.clone(),
            capabilities,
            claims: leaf.facts.clone(),   // leaf facts → claims (co-design seam #1)
            verified_at_secs: opts.now_secs,
        })
    }
}
```

> **Shape reconciliation (co-design seam #1 — read before coding).** task.md
> and M04B's task.md both describe the object B1 produces loosely as
> `{caller_did, capabilities, scopes, claims, roles, env}`. The **authoritative
> type is ADR-0015's** `SessionContext { subject_did, capabilities, claims,
> verified_at_secs }` (already shipped in B0). B1 **deliberately folds** the
> prose's extra fields, it does not add them:
> - `caller_did` → `subject_did` (rename only).
> - `scopes` and `roles` → `capabilities`. In this UCAN model a "scope"/"role"
>   is not a distinct primitive: it *is* a set of `Capability { with, can }`
>   grants. M04B's compiler binds capabilities (resource + ability) directly as
>   `?` params; a separate `scopes`/`roles` bag would be redundant surface with
>   no verified backing. Fail-closed authority lives only in `capabilities`.
> - `env` and any non-capability claim → `claims` (the token's `facts` map).
>   `env`-style attributes (request-time context for stage-4 ABAC) are just
>   claims; M04B reads them from `claims`.
>
> **This is a decision, not a gap** — `SessionContext` stays the 4-field
> ADR-0015 type; no `scopes`/`roles`/`env` fields are added in M04A. If M04B's
> compiler genuinely needs a field that cannot be expressed as a capability or
> a claim, that is an ADR-0015 amendment to raise in the D-04-01/D-04-02
> co-design, not a silent B1 addition.

### 2.4 `Capability::covers` (`crates/ucan/src/capability.rs`)
Attenuation between two capabilities — parent covers child iff same-or-broader
resource **and** the ability entails. Factor the resource-scope rule already
inlined in `Capability::grants` (capability.rs:98-103) so both share it:
```rust
impl Capability {
    /// Whether `self` (a held/parent capability) authorizes everything `other`
    /// (a requested/child capability) asks for. A `substrate:`-scoped `self`
    /// covers any resource; otherwise resources must match exactly.
    pub fn covers(&self, other: &Capability) -> bool {
        (self.with.is_substrate_scope() || self.with == other.with)
            && self.can.entails(&other.can)
    }
}
// `grants` becomes: self.covers(&Capability{ with: resource.clone(), can: ability.clone(), caveats: None })
// — or keep grants as-is and just add covers; grants already encodes the same
//   rule, so refactoring is optional (cleaner but not required).
```

### 2.7 Unit tests (`token.rs`, in-crate) — the crux of B1
Build tokens with real `Identity`s (`owner`, `alice`, `bob`). `admin_root =
derive_did_key(owner.public_key())`; `is_trusted_root = |iss, _| iss ==
admin_root`.

- **happy path, direct root:** `owner` issues to `alice` `[data-layer/read on
  synapp:app1:svc:s1]`; `verify_chain` (audience=alice) returns that one cap.
- **happy path, one-hop attenuation:** `owner`→`alice` `[data-layer/admin]`,
  then `alice`→`bob` `[data-layer/write]` with the first as its proof;
  verify(leaf=bob-token, audience=bob) returns `[data-layer/write]`.
- **escalation blocked:** `alice` (holding `read`) issues `bob` `[write]`;
  `covers` fails → `write` dropped (empty result). Fail-closed.
- **untrusted root dropped:** `alice` self-issues to `bob` with no proof and
  `alice != admin_root` → empty result.
- **audience mismatch:** leaf addressed to `alice`, presented as `bob` →
  `Err`.
- **expiry / not-yet-valid:** expired leaf → `Err`; leaf with a proof that is
  expired → `Err` (proof `verify_self` aborts).
- **tampered signature** (any token in the chain) → `Err`.
- **tampered capability after signing** (mutate `capabilities` post-issue) →
  signature check `Err`.
- **continuity break:** proof `audience != child issuer` → that proof
  contributes nothing (child cap dropped unless independently rooted).
- **substrate-scope covers any resource:** `owner` issues `bob`
  `[substrate/admin on substrate:<node>]`; `bob` then asks (via
  `SessionContext::has_capability`) for `data-layer/admin` on an arbitrary
  service resource → granted (exercises `covers` substrate-scope arm).
- **`from_verified_chain`:** populates `subject_did`/`capabilities`/`claims`
  (from `facts`)/`verified_at_secs`.
- **`DidKeyNormalizer`:** passes a real did:key, rejects `did:web:...`.

### 4.2 `build_caller` verifies the chain (`crates/router/src/route_handler/io.rs:47`)
Extend the existing `build_caller` (keep the B0 admin-root direct-equality
grant; add the chain path). New signature — it needs the resolver for
revocation and must be `async`:

```rust
async fn build_caller(
    preamble: &RoutePreamble,
    id: &VerifiedIdentity,
    admin_root: Option<&str>,
    resolver: &dyn MasterAnchorResolver,
) -> CallerContext {
    let now = now_secs();
    let mut session = SessionContext {
        subject_did: id.master_did.clone(),
        verified_at_secs: now,
        ..Default::default()
    };
    let mut auth = AuthLevel::Delegated;

    // B0 path (kept): the substrate owner's own DID gets substrate/admin.
    if admin_root == Some(id.master_did.as_str()) {
        session.capabilities.push(Capability {
            with: ResourceUri::substrate(&id.master_did),
            can: Ability(Ability::SUBSTRATE_ADMIN.to_string()),
            caveats: None,
        });
    }

    // B1 path: verify a presented UCAN chain rooted at the node admin root,
    // addressed to this verified connection identity.
    if let (Some(token), Some(root)) = (&preamble.ucan, admin_root) {
        let is_root = |iss: &str, _res: &ResourceUri| iss == root;
        let opts = ChainVerifyOpts {
            expected_audience_did: &id.master_did,
            is_trusted_root: &is_root,
            now_secs: now,
        };
        match SessionContext::from_verified_chain(token, &opts) {
            Ok(verified) if ucan_chain_not_revoked(token, resolver).await => {
                // Merge verified capabilities/claims; upgrade auth level.
                session.capabilities.extend(verified.capabilities);
                for (k, v) in verified.claims { session.claims.insert(k, v); }
                auth = AuthLevel::Ucan;
            }
            Ok(_) => tracing::warn!("UCAN chain rejected: a chain DID is revoked"),
            Err(e) => tracing::warn!("UCAN chain verification failed: {e}"),
            // Fail-open to Delegated here is deliberate: a bad *authorization*
            // token does not sink an otherwise-verified *transport* identity;
            // the caller simply holds no UCAN capabilities. The admin/native
            // gates then fail closed downstream. (A malformed *delegation*
            // cert is still a hard reject in `handle_stream`, unchanged.)
        }
    }

    CallerContext {
        caller_did: id.master_did.clone(),
        app_instance: None,
        session,
        auth,
        proof: Some(CallerProof {
            pubkey_hex: preamble.pubkey.clone().unwrap_or_default(),
            delegation_json: preamble.delegation.as_ref().and_then(|c| c.to_json().ok()),
            // ucan_json: see Flag F5 — NOT added at B1.
        }),
    }
}
```

**Revocation reuse** (`ucan_chain_not_revoked`, new fn in io.rs): faithful
generalization of the delegation-cert revocation already in `handshake.rs:74-84`
— for each edge in the token tree, resolve the *issuer's* master anchor and
reject if the *audience* DID is in its `revoked_keys`. Add
`CapabilityToken::chain_edges(&self) -> Vec<(&str, &str)>` to the ucan crate
(pre-order walk yielding `(issuer_did, audience_did)`), then:
```rust
async fn ucan_chain_not_revoked(token: &CapabilityToken, resolver: &dyn MasterAnchorResolver) -> bool {
    for (issuer, audience) in token.chain_edges() {
        match resolver.resolve_master_anchor(issuer).await {
            Ok(anchor) if anchor.revoked_keys.iter().any(|k| k == audience) => return false,
            Ok(_) => {}
            Err(_) => {}   // anchor unresolvable → treat as not-revoked, matching
                           // the delegation path's behavior (it errors the whole
                           // verify only on timeout, not on a missing anchor for a
                           // key that isn't revoked). See Flag F3.
        }
    }
    true
}
```

(io.rs:229). The UCAN is verified once, in `build_caller`, before either
transport arm. So the bridge inherits UCAN capabilities for free.
> Flag F2: HTTP clients presenting a UCAN via an `Authorization:` header (rather
> than the `ucan=` preamble param) is a natural future ergonomic, but the
> preamble is parsed *before* the HTTP request in this codebase, so the header
> path would need a second verify site in `http.rs`. Out of B1 scope — B1's one
> verify site (`build_caller`) covers both transports via the preamble.

## 7. Integration test — reference scenario step 21

New `crates/router/tests/ucan_context.rs` (pattern: `native_dispatch_identity.rs`):

1. Generate `owner` (= configured `admin_ucan_root`) and `client` identities.
2. `owner` issues a `CapabilityToken` to `client`'s DID granting
   `data-layer/read` (or `data-layer/admin` to also exercise the DDL gate) on a
   specific `synapp:app:svc` resource, `expires_in` 1 h, no proofs.
3. Build a `RoutePreamble` with `pubkey = client.public_key()` and
   `ucan = Some(token)`; run it through `handle_stream` /
   `dispatch_json_rpc_once` against a recording `NativeService` (as the B0 test
   does), with `RouteHandlerInner.admin_ucan_root = Some(owner_did)` and a
   `MockResolver` returning an empty (non-revoking) anchor.
4. **Assert:** the `NativeService` receives a `CallerContext` whose
   `session.has_capability(resource, data-layer/read)` is `true`, `auth ==
   AuthLevel::Ucan`, and `subject_did == client_did`. (This *is* step 21:
   "client presents a UCAN → node verifies the chain → normalized capability
   reaches dispatch.")
5. **Negative:** same token but presented by a *different* pubkey (audience
   mismatch) → capability absent, `auth == Delegated`. And: a token rooted at a
   non-admin issuer → capability absent.
6. **Revocation:** anchor with the client DID in `revoked_keys` → capability
   absent (chain rejected).

Optionally add a full-substrate e2e variant later; the dispatch-level test
above is the authoritative proof for the exit criterion (matches how B0's
"single most important test" is a dispatch-level integration test, not
Playwright).

## 8. Ambiguities / stale-anchor flags (resolve before/while coding)

- **F1 — Trust root for *service* resources is unresolved in the task.** ADR-0015
  §2 says the trusted root is "the resource owner's DID for service
  capabilities." But no owner DID is recorded anywhere until Slice **B7**
  (task.md:356-407 explicitly: `catalog.rs` "records no owner/creator"). **B1
  decision (this plan):** the node admin root is the sole trusted issuer for all
  resources; owner-rooted chains await B7. This is coherent (`substrate/admin ⊇
  everything on that node`) and makes B1 a clean generalization of B0. **Confirm
  this scoping is acceptable** — the alternative (block B1 on B7) contradicts
  the task's own slice order (B1 before B7).
- **F2 — "Gateway verifies the UCAN chain at ingress" names the wrong
  component.** In this codebase `client_gateway` is a byte tunnel that only
  sniffs `Host:` (system-architecture.md:1937; B0 confirmed). The real ingress
  verify site is the **router** (`build_caller`). B1 wires it there. The
  spec/task "gateway" wording is a vocabulary mismatch, not a missing component.
- **F3 — Revocation semantics for UCAN are under-specified.** ADR-0015 §2 says
  "reuses the existing DHT-revocation check." The existing check is DID-in-anchor
  (`handshake.rs:80`). B1 generalizes it per §4.2 (each issuer's anchor, audience
  membership). **Per-token-id revocation** (revoking one specific token without
  revoking the audience DID) is **deferred** — no token-id revocation registry
  exists. Unresolvable anchors are treated as not-revoked (matching the
  delegation path, which only hard-fails on *timeout*). Flag if stricter
  fail-closed-on-unresolvable is wanted.
- **F4 — `AuthNormalizer` has no consumer at B1** (§2.5). Shipped as the
  ADR-mandated seam only; did:key normalization is a no-op given the handshake
  already yields did:key. Not integration-wired.
- **F5 — Cross-node UCAN forwarding is out of B1 scope.** `CallerProof`
  (native.rs:46) carries only `pubkey_hex`/`delegation_json`. Forwarding a UCAN
  across a proxy hop (needed by M04B's B3 federated fetch, ADR-0016 §6) means
  adding `ucan_json` to `CallerProof` and re-presenting it on the outbound
  preamble in `ProxyRouter`. B1 is **ingress-only**; A1's status already records
  `CallerProof` "carries only the delegation half per the plan's own TODO."
  Adding `ucan_json` is a small, additive B3 follow-on — noted, not built here.
- **F6 — Stale preamble doc-comment.** preamble.rs:79-88's "Authentication
  status, as of M3B/M3C" block already went stale at B0 (verify is no longer
  conditional). B1 touches this file; refresh that block to note UCAN capability
  extraction (optional cleanup, low priority — B0 didn't).
- **F7 — `proofs`-excluded-from-signature is a deliberate design choice** (§2.2),
  matching the `DelegationCertificate` model and UCAN semantics (each proof
  self-signs; continuity binds). If a reviewer expects the leaf signature to
  cover the whole chain, point them here — the security argument is in the
  `CapabilityToken` doc-comment.
