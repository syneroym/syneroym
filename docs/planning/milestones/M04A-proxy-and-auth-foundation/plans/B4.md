# Slice B4 — `AggregationPipeline` — Implementation Plan

> **Requirement:** `[PLT-DAT]`; closes M04A gate item #2.
> **Depends on:** nothing (task.md:337 "Independent — no auth dependency; may
> start any time"). It does, however, **reuse B5's already-shipped machinery**:
> the `sql-value` variant, the `raw-query-result` record, the
> `wit_to_rusqlite_value`/`rusqlite_to_wit_value` converters, and the reader-pool
> `run_query_raw` executor. B5 is merged, so all of these exist today.
> **ADR:** [ADR-0007](../../../decisions/0007-data-layer-wit-interface.md)
> (Accepted) — its §"AggregationPipeline" defers this work to M4 and says it
> "should translate MongoDB aggregation-pipeline stages to SQLite constructs
> (`GROUP BY`, `HAVING`, views)". This slice **amends ADR-0007** to record the
> three shape decisions below (§0.1).

## 0. Summary of the change

Add an **`aggregate`** capability to the `data-layer` `store` interface: a caller
supplies a target collection plus a JSON *aggregation document* (a single object
with `$match`/`$group`/`$having`/`$project`/`$sort`/`$limit` keys); the host
compiles it to a **parameterized** SQLite `SELECT … GROUP BY … HAVING …`
statement and returns a `raw-query-result { columns, rows }`. It is the
`GROUP BY`/`HAVING`/projection counterpart to the safe JSON *filter* DSL
(ADR-0007) — same "MongoDB-style JSON → parameterized SQLite" philosophy, same
safe-by-construction whitelist, **no privileged gate**. It is *not*
`query-raw` (B5): `query-raw` takes arbitrary caller SQL and is Admin-gated;
`aggregate` takes a whitelisted operator document and is open to any guest,
exactly like `query`.

Surfaces touched, in dependency order:

1. **WIT** (`data-layer.wit`, both copies) — one new function, `aggregate`.
   **No new types** — it reuses `raw-query-result`/`sql-value` shipped by B5.
2. **`syneroym-data-db`** — a new `aggregate` module (the pipeline→SQL compiler),
   a `do_aggregate` reader-pool helper, a `ServiceStore::aggregate` trait method
   + its two impls.
3. **`syneroym-sandbox-wasm`** — guest-side `store::Host::aggregate` (mirrors
   `query`; **no** capability gate).
4. **`syneroym-control-plane`** — native-side `aggregate` arm in
   `dispatch_data_layer`, reusing B5's `RawQueryResultDto` encoding (lifted to a
   shared helper).
5. **Tests** — compiler unit tests (data-db `aggregate` module), end-to-end
   store tests (data-db `tests_crud.rs`), native-dispatch test
   (`router`/`control_plane`).

No config change. No writer-loop / KEK / DB-isolation change. Additive minor WIT
bump — existing guests unaffected, `wasm32-wasip2` stays buildable.

### 0.1 Confirmed design decisions (requester, 2026-07-16) — amend ADR-0007

Three shape decisions were confirmed before planning (following B5's precedent of
confirming return-shape/DSL choices with the requester rather than guessing):

- **D1 — Separate `aggregate` function returning `raw-query-result`, not an
  extension of `query`.** task.md:337 and ADR-0007's prose both say "on the
  `query` function", but `query` returns `query-result` = `list<record-read-value>`,
  a **fixed** `{id, payload, creator-id, created-at, updated-at}` shape. A
  grouped/projected result (`SELECT category, count(*) …`) cannot fit it — this
  is the identical mismatch B5 documented for `query-raw` (B5.md §0.1 D1) and
  resolved with `raw-query-result`. **Decision:** a distinct
  `aggregate: func(collection, pipeline) -> result<raw-query-result, …>`,
  reusing B5's `raw-query-result`. Amend ADR-0007 to record this and why
  extending `query` was rejected.
- **D2 — DSL is a single JSON object, not an ordered pipeline array.** The
  document has optional keys `$match`, `$group` (required), `$having`,
  `$project`, `$sort`, `$limit`, compiled in one deterministic pass. This is
  narrower than Mongo's ordered `[{…},{…}]` pipeline but covers exactly the
  operators task.md names (`$group`/`$having`/projections) and is far simpler to
  compile and validate. Amend ADR-0007's "translate aggregation-pipeline stages"
  wording to name this concrete surface.
- **D3 — Physical collections only; init-defined logical views deferred.**
  task.md:337 says aggregation "can target physical collections and init-defined
  logical views". Views are **deferred** (Flag F1): field access assumes the
  physical `{id, payload}` row shape (`json_extract(payload, '$.field')`), which
  does not hold for an arbitrary `CREATE VIEW`. Record the deferral in ADR-0007
  and `status.md`.

### 0.2 Forward seam for M04B FDAE (R1.1) — record, don't close

`aggregate` is a **second, independent read path into `payload`** that does not
flow through `query`'s compiler. M04B's FDAE RLS/CLS *pushdown sieve* is scoped
in [M04B/task.md](../../M04B-fdae-policy/task.md:266) specifically to
`data-layer::query` (reference-scenario step 22: "*A `data-layer::query` call is
transparently filtered by FDAE's SQL pushdown sieve*"). Unless M04B also wraps
`aggregate`, a caller who is row-restricted on `query` post-M04B could read the
same rows in aggregate form via this path — a **policy-bypass gap**.

M04A's task.md has a "Relationship to M04B" section that calls out exactly this
kind of forward seam for B1 (SessionContext) and A1 (proxy shape); B4 was not
given one. **This slice does not close the gap — FDAE is M04B — but it must
record it** so M04B's design accounts for `aggregate` as a sieve target from day
one. Two things, both doc-only for B4:
- Add a sentence to ADR-0007's amendment: `aggregate` shares `query`'s trust
  level and, like `query`, is a **future FDAE-sieve target** — M04B must wrap
  both, not just `query`.
- Add a "Forward seam / M04B" note to the B4 `status.md` section, and (at
  milestone-close doc pass) a line to M04A task.md's Relationship-to-M04B
  section naming `aggregate` alongside `query`.

The single-object DSL (D2) actually *helps* here: `$match` already compiles
through `filter::compile_filter`, the same seam M04B's sieve will hook, so an
injected RLS predicate can be `AND`-ed into `aggregate`'s `WHERE` by the same
mechanism M04B uses for `query` — the gap is one of *remembering to wire it*,
not of structural incompatibility. Note this in status.md so M04B inherits the
observation.

## 2. `syneroym-data-db`

### 2.1 New module: `crates/data_db/src/aggregate.rs` (the compiler)

Pure and DB-free — mirrors `filter.rs`'s posture (produces SQL text + bound
params, interpolates nothing caller-supplied). Add `pub mod aggregate;` to
`crates/data_db/src/lib.rs` (next to `pub mod filter;`, line 2).

```rust
//! MongoDB-style JSON aggregation document -> parameterized SQLite
//! `SELECT ... GROUP BY ... HAVING ...` compiler (ADR-0007, Slice B4).
//!
//! Pure and DB-free. Field paths and all literal values are bound as `?`
//! placeholders; the only caller-derived text ever interpolated into SQL is
//! output *aliases* and *field names*, each first passed through
//! `validate_identifier` (SQL-identifier charset only) -- never a value, never
//! an operator.

use rusqlite::types::Value;
use serde_json::Value as Json;

use crate::filter;               // reuse compile_filter for $match
use crate::host_store::DataLayerError;
use crate::sqlite::validate_identifier;   // NOTE: make this `pub(crate)` -- see §2.2

/// A compiled aggregation: the full SQL statement plus its bound parameters in
/// binding (left-to-right textual) order.
pub struct CompiledAggregation {
    pub sql: String,
    pub params: Vec<Value>,
}
```

**Grammar accepted** (reject everything else with
`DataLayerError::SchemaViolation(msg)`):

| Key         | Shape                                              | Compiles to |
|-------------|----------------------------------------------------|-------------|
| `$match`    | a MongoDB filter document (same as `query.filter`) | `WHERE <compile_filter>` |
| `$group`    | **required.** `{ "_id": <key>, "<out>": {"<$acc>": <arg>}, … }` | `SELECT`-list + `GROUP BY` |
| `$having`   | predicate over `$group` output aliases             | `HAVING …` |
| `$project`  | `["out1","out2", …]` (subset/reorder of output cols) | outer `SELECT` wrap |
| `$sort`     | `{ "<out>": 1 \| -1 }`                              | `ORDER BY <out> ASC\|DESC` |
| `$limit`    | positive integer                                   | `LIMIT ?` (min'd with `MAX_QUERY_PAGE_SIZE`) |
| `$skip`     | non-negative integer                               | `OFFSET ?` — group pagination past the page cap |

`$group` details:
- **`_id`**: either a **string field path** (`"category"` or dot-path
  `"addr.city"`) → grouping key, emitted as
  `json_extract(payload, ?) AS _id` (param `$.category`), with `GROUP BY _id`;
  **or** JSON `null` → no `GROUP BY` (whole collection = one group), no `_id`
  output column. A composite-key object (`{"c":"category","r":"region"}`) is
  **rejected** with `schema-violation("composite $group._id is not supported")`
  — Flag F2 (possible extension).
- **Empty-output guard (R2.3):** the SELECT list must be non-empty. If `_id` is
  `null` **and** there are zero accumulators, `select_exprs` would be empty and
  assembly would emit the syntactically invalid `SELECT  FROM <collection>`.
  **Reject** this up front with
  `schema-violation("$group must define _id or at least one accumulator")`. (A
  non-null `_id` with zero accumulators is *valid* — it returns distinct group
  keys, `SELECT json_extract(payload, ?) AS _id … GROUP BY _id`.)
- **accumulators** `"<out>": {"<$acc>": <arg>}`, one operator key each:
  - `{"$sum": 1}` (numeric literal `1`) → `COUNT(*) AS <out>` (no param). Any
    other numeric literal → `schema-violation` (avoids surprising `SUM(const)`
    semantics; Flag F3).
  - `{"$sum": "field"}`  → `SUM(json_extract(payload, ?)) AS <out>`, param `$.field`.
  - `{"$avg": "field"}`  → `AVG(json_extract(payload, ?)) AS <out>`.
  - `{"$min": "field"}`  → `MIN(json_extract(payload, ?)) AS <out>`.
  - `{"$max": "field"}`  → `MAX(json_extract(payload, ?)) AS <out>`.
  - any other `$acc` → `schema-violation("unsupported accumulator: <op>")`.
- Every `<out>` alias **and** the literal `_id` alias are validated with
  `validate_identifier` before interpolation (they appear as `AS <out>`).
- **Payload-only field access (Flag F8, R2.1).** `_id`, accumulator args, and
  `$match` fields are *all* resolved as `json_extract(payload, '$.field')` —
  the physical columns (`id`, `creator_id`, `created_at`, `updated_at`) are
  **not** reachable, so e.g. "count per `creator_id`" is not expressible. This
  is **deliberate consistency with the existing `query` DSL, not a narrowing of
  it**: `filter.rs` also treats every field as a payload path and has *zero*
  physical-column awareness (verified — it unconditionally emits
  `json_extract(payload, ?)` at `filter.rs:197,224,239,249,253`). A reviewer read
  `$match`'s `filter.rs` reuse as implying physical-column awareness; there is
  none to inherit. Adding it to `aggregate` alone would make `aggregate` able to
  group by `creator_id` while `query` still cannot *filter* by it — an
  asymmetry that violates the "same DSL family" framing. **Recommendation:** keep
  B4 payload-only (consistent), and if host-column access is wanted, add a
  bounded allowlist for the four host-injected columns to **both** `filter.rs`
  and `aggregate.rs` together in a follow-on, so the two stay symmetric. Flagged,
  not silent.
- Output **column order** absent `$project`: `_id` first (if present), then
  accumulators in `serde_json::Map` iteration order. Because the workspace does
  **not** enable serde_json `preserve_order`, that map is a `BTreeMap` → order is
  **alphabetical and deterministic** (verified: no `preserve_order` feature in
  any `Cargo.toml`). Tests can rely on this; callers wanting explicit order use
  `$project`. Document in the WIT doc-comment. Flag F4.

`$having` compiler (small, local to this module — do **not** reuse
`filter::compile_filter`, which hard-codes `json_extract(payload, ?)`):
- Accepts `{ "<alias>": <scalar> }` (→ `<alias> = ?`),
  `{ "<alias>": {"$gt|$gte|$lt|$lte|$ne": <scalar>} }` (→ `<alias> <op> ?`), and
  top-level `$and`/`$or` arrays of such (→ `( … AND/OR … )`).
- **`<alias>` must be a member of the `$group` output-alias set** (`_id` + the
  accumulator names) — reject unknown aliases with
  `schema-violation("$having references unknown field: <alias>")`. It is emitted
  as a **bare validated identifier**, never `json_extract`. Values bound as `?`.
- This membership check is the injection defense for `$having` (the alias is the
  only identifier interpolated; values are always bound).
- **Depth guard (R1.3).** This is a *second, independent* recursive compiler
  (`$match` reuses `filter::compile_filter`, which already enforces
  `MAX_FILTER_DEPTH = 10` at `filter.rs:57,132`; `$having` does **not** inherit
  it). ADR-0007's Consequences require every JSON query compiler to bound
  nesting. Thread a `depth: u32` through the `$having` `$and`/`$or` recursion and
  reject `> 10` levels with `schema-violation("$having document too deeply
  nested")`, reusing the same constant/limit as `filter.rs` (import
  `filter::MAX_FILTER_DEPTH` — bump it to `pub(crate)`, or define a local
  `MAX_HAVING_DEPTH = 10` mirroring it). Add a `having_nested_over_depth_rejected`
  unit test (§5.1).

**Assembly** (produces `CompiledAggregation`; push params in exact textual
order):

```
inner_select = "SELECT " + join(select_exprs, ", ")            // _id + accumulators
inner        = inner_select + " FROM " + collection            // collection pre-validated by caller (§2.2)
             + where_sql?      // "WHERE " + $match clause         (params: match)
             + group_by_sql?   // "GROUP BY _id"  (only if _id is a field path)
             + having_sql?     // "HAVING " + $having clause        (params: having)

if $project or $sort or $limit or $skip present:
    sql = "SELECT " + projected_cols_or_star + " FROM (" + inner + ")"
        + order_by_sql?        // "ORDER BY <alias> ASC|DESC[, …]"  (no params)
        + limit_offset_sql?    // see below                          (params: limit?, skip?)
else:
    sql = inner

// param order == textual order:
params = select_path_params ++ match_params ++ having_params ++ limit_param? ++ skip_param?
```

`$limit`/`$skip` → `limit_offset_sql`:
- `$limit L`, no `$skip` → `LIMIT ?` (bind `min(L, MAX_QUERY_PAGE_SIZE)`).
- `$skip S` (with or without `$limit`) → `LIMIT ? OFFSET ?` — SQLite requires a
  `LIMIT` before `OFFSET`, so when `$skip` is present but `$limit` is not, bind
  `LIMIT` to `MAX_QUERY_PAGE_SIZE` and `OFFSET` to `S`. This is what lets a
  caller page past the `run_query_raw` cap: page *k* is
  `{"$skip": k*pagesize, "$limit": pagesize}` (R2.2). Param order is `limit`
  then `skip`, matching the `LIMIT ? OFFSET ?` textual order.

Notes:
- `select_path_params` are the `$.field` paths bound by `_id` and the
  accumulators, in emission order — they appear textually first (in the SELECT
  list), so they bind first.
- `GROUP BY _id` / `ORDER BY <alias>` reference **output aliases** (SQLite
  permits this), so they add **no** params and cannot duplicate the path bindings.
- `$project` columns must each be in the output-alias set (validated +
  membership-checked), interpolated as bare identifiers in the outer `SELECT`.
- **Determinism nicety (recommended):** when grouping by a key and `$sort` is
  absent, append a default `ORDER BY _id ASC` so row order is stable for tests
  and callers. Flag F5 (drop if undesired).

Compiler entry point:

```rust
pub fn compile(collection: &str, pipeline_json: &str)
    -> Result<CompiledAggregation, DataLayerError>
{
    let doc = parse_object(pipeline_json)?;              // schema-violation on non-object/invalid JSON
    // reject unknown top-level keys up front:
    //   allowed = {$match,$group,$having,$project,$sort,$limit}
    let group = doc.get("$group").ok_or_else(|| schema("aggregate requires a $group stage"))?;
    // ... build select_exprs, aliases set, select_path_params from $group ...
    // ... where_sql/match_params  from filter::compile_filter($match)   (reused verbatim)
    // ... having_sql/having_params from local having compiler (alias-checked)
    // ... project / sort / limit ...
    Ok(CompiledAggregation { sql, params })
}
```

> `filter::compile_filter` takes `Option<&str>`, but `$match` here is a
> `serde_json::Value`. Cheapest reuse: `serde_json::to_string(&match_val)` then
> `compile_filter(Some(&s))`. Confirm `compile_filter` is `pub` (it is —
> `filter.rs:28`). Alternatively expose a `compile_document`-level entry that
> takes a `&Map`; **default to the `to_string` round-trip** to avoid touching
> `filter.rs`'s API.

### 2.2 `do_aggregate` reader-pool helper (`crates/data_db/src/sqlite.rs`)

Add next to `do_query` (~line 372). It compiles, then **reuses B5's
`run_query_raw`** (line 508) verbatim for execution — that function already
prepares, asserts `stmt.readonly()`, extracts `columns`, iterates rows through
`rusqlite_to_wit_value`, and enforces the `MAX_QUERY_PAGE_SIZE` → `QuotaExceeded`
cap. The aggregation SQL is entirely host-generated (only bound params +
validated identifiers), so it is `readonly()` by construction and needs **no**
authorizer/progress-handler wrap (those defend against *arbitrary caller SQL*,
which `aggregate` never accepts).

```rust
fn do_aggregate(
    conn: &Connection,
    collection: &str,
    pipeline_json: &str,
) -> Result<host_store::RawQueryResult, host_store::DataLayerError> {
    validate_identifier(collection)?;                 // same guard do_query uses
    let compiled = aggregate::compile(collection, pipeline_json)?;
    // `run_query_raw` asserts readonly() + caps rows; our SQL is always a SELECT.
    run_query_raw(conn, &compiled.sql, &compiled.params)
}
```

**Required visibility change:** `validate_identifier` (`sqlite.rs:58`) and
`run_query_raw` (`sqlite.rs:508`) are currently private module fns. `aggregate.rs`
calls `validate_identifier`; `do_aggregate` (in `sqlite.rs`) calls both — same
module, no change needed for `run_query_raw`. For `aggregate.rs` to call
`validate_identifier`, mark it **`pub(crate)`** (`fn` → `pub(crate) fn`). No
other visibility change.

> `aggregate::compile` takes `collection` only to interpolate `FROM <collection>`;
> `do_aggregate` still calls `validate_identifier(collection)` itself so the guard
> is co-located with the other reader helpers and not solely inside the compiler.
> (Belt-and-suspenders; the compiler also validates before interpolating.)

**Refactor to share the encoder** (avoid duplicating the DTO + row-mapping):
lift `SqlValueDto` and `RawQueryResultDto` out of the `query-raw` arm to
**module scope**, and add a module-level helper:

```rust
// module scope (near the other DTOs, ~line 172)
#[derive(serde::Serialize, serde::Deserialize)]
#[serde(tag = "type", content = "value", rename_all = "snake_case")]
enum SqlValueDto { Text(String), Integer(i64), Real(f64), Boolean(bool), Null }

#[derive(serde::Serialize)]
struct RawQueryResultDto { columns: Vec<String>, rows: Vec<Vec<SqlValueDto>> }

fn raw_query_result_payload(result: host_store::RawQueryResult) -> RpcResult<Payload> {
    let dto = RawQueryResultDto {
        columns: result.columns,
        rows: result.rows.into_iter().map(|row| {
            row.into_iter().map(|v| match v {
                SqlValue::Text(s) => SqlValueDto::Text(s),
                SqlValue::Integer(i) => SqlValueDto::Integer(i),
                SqlValue::Real(f) => SqlValueDto::Real(f),
                SqlValue::Boolean(b) => SqlValueDto::Boolean(b),
                SqlValue::Null => SqlValueDto::Null,
            }).collect()
        }).collect(),
    };
    to_payload(&dto)
}
```

Then the **`query-raw` arm** (lines 363-414) drops its inline `SqlValueDto`/
`RawQueryResultDto`/mapping and ends with `raw_query_result_payload(result)`
(its own `SqlValueDto` *request*-decode for `params` stays — that decode is
query-raw-specific; only the *result* encode is shared). The new arm:

```rust
"aggregate" => {
    #[derive(serde::Deserialize)]
    struct Req { collection: String, pipeline: String }
    let req: Req = parse_params(&invocation)?;
    let result = store
        .aggregate(&req.collection, &req.pipeline)
        .await
        .map_err(data_layer_error)?;
    raw_query_result_payload(result)
}
```

> Confirm `RpcResult`/`Payload` are the exact return alias/type `to_payload`
> yields in this file (check the existing `fn to_payload` signature and the
> `dispatch_data_layer -> RpcResult<NativeResponse>` chain, line 163). If the
> helper's return type is awkward, inline the encoder as a closure instead —
> the goal is not re-duplicating the 15-line row map, not the exact helper
> signature. `SqlValue`/`host_store` are already imported (line 38 / via
> `data_layer_error` usage). **No new import** beyond what query-raw already
> pulls in.
>
> **No capability gate** in this arm — contrast the `execute-ddl`/`query-raw`
> arms, which gate on `data-layer/admin`. `aggregate` is deliberately open, like
> `query` (line 243). This is the native-side proof of D2's "safe DSL, no gate".
>
> **On the alleged `SqlValue` vs `SqlValueDto` collision (R2.4):** there is none.
> `SqlValue` is the imported alias for `host_store::SqlValue` (`synsvc_native.rs:38`)
> and `SqlValueDto` is a distinct type name — matching `SqlValue::Text(s) =>
> SqlValueDto::Text(s)` is unambiguous. This exact match already lives in B5's

## 7. Flags — ambiguity / staleness / risk

- **F0 — task.md "on the `query` function" is imprecise / ADR-0007 says the same.**
  Both predate B5's `raw-query-result` decision. Resolved by D1: a separate
  `aggregate` function. Amend ADR-0007 (§0.1); note in `status.md` that "on
  `query`" was read as "on the `query` *capability/DSL family*", not literally
  the `query` WIT function.
- **F1 — init-defined logical views deferred (D3).** task.md:337 lists them;
  this slice does physical collections only (field access hard-assumes the
  `{id, payload}` row shape). Record the deferral explicitly in ADR-0007 +
  `status.md`; a follow-on can add view support via `PRAGMA table_info`
  introspection (bare columns vs `json_extract`).
- **F2 — composite `$group._id` rejected.** Single field path or `null` only.
  Multi-field grouping is a clean future extension; rejecting loudly (not
  silently mis-grouping) is the safe interim.
- **F3 — `$sum` numeric literal.** Only `{"$sum": 1}` (→ `COUNT(*)`) is
  accepted; other numeric literals are rejected to avoid ambiguous
  `SUM(constant)` semantics. `$sum` of a field sums it. Document in the ADR.
- **F4 — output column order** absent `$project` is alphabetical (BTreeMap),
  not pipeline-insertion order (Mongo would preserve insertion). Deterministic,
  documented; `$project` gives explicit control. Note in the WIT doc-comment.
- **F5 — default `ORDER BY _id ASC`** when grouping without `$sort` (§2.1). A
  determinism nicety; if the requester prefers "no implicit ordering", drop it
  and make tests assert set-wise. **Recommend keeping it.**
- **F6 — page-cap test cost** (§5.2). Seeding >1000 rows to force
  `QuotaExceeded` is slow; acceptable to lean on B5's existing `run_query_raw`
  cap unit test and assert the wiring more cheaply. Decide when coding.
- **F7 — `$having` without `$group` grouping key** (`_id: null`): SQLite treats
  the whole table as one group, so `HAVING` on an accumulator alias still works.
  Edge case; a unit test can pin it, else document.
- **F8 — payload-only field access (R2.1)** (§2.1). `_id`/accumulators/`$match`
  resolve only `payload` JSON paths, never the physical columns — consistent
  with `query`'s filter DSL (which has no physical-column awareness), *not* a
  narrowing of it. Recommendation: if host-column grouping is later wanted, add a
  four-column allowlist to `filter.rs` **and** `aggregate.rs` together. Flagged
  known limitation for B4.
- **F9 — FDAE forward seam (R1.1)** (§0.2). `aggregate` is a second payload-read
  path M04B's `query`-scoped pushdown sieve won't cover unless told to. Doc-only
  for B4: record in ADR-0007 + status.md so M04B wraps both paths.
