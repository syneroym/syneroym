## 0. Summary and recommended scope split

task.md's B7 bundles two separable things: **recording who owns what**
(attribution) and **enforcing who may deploy** (authorization). They have
different dependencies, different risk, and only the first is a prerequisite
for the other. Recommend shipping them as two PRs:

| | **B7a — Ownership attribution** | **B7b — Deploy grant** |
|---|---|---|
| task.md items | 2, 3, 5 (**4 dropped — F9**) | 1 |
| Delivers | owner recorded per deployed service; `list` filtered by owner; the unowned-substrate posture made explicit | revocable, per-grantee, non-re-delegatable `orchestrator/deploy` grant; deploy / undeploy / per-service status-check gated; Tier 1 for orchestrator |
| Needs | nothing new in `syneroym-ucan` | ADR-0015 A1/A3/A4/A6 implemented in `syneroym-ucan` |
| Unblocks | **`owner_of(resource)`** — which A6 needs and `session.rs`'s `TODO(B7)` guards | `roymctl` acting as a real principal |
| Risk | low — additive; admission unchanged | high — can deny the operator out of their own substrate |

**Why this split and not "all of B7 at once":** A6's resource-scoped
`is_trusted_root` is *defined in terms of* `owner_of(resource)`, and `owner_of`
does not exist until B7a's owner store does. Doing them together means writing
the capability-model change against a store being designed in the same PR. B7a
is also independently valuable: it closes the concrete gap task.md actually
cites (`list` returns every deployed app to any caller) without touching the
admission path at all.

**The rest of this plan assumes the split.** §2 is B7a (executable once §1's
flags are resolved), §3 is B7b.

### 0.1 The organizing idea (post-`f95206b`)

`f95206b` ("docs: state per-layer defaults and anonymous access") landed after
this plan was first drafted and **changed its shape**, for the better. Three
things from it govern the design below:

1. **"The grant layer is default-deny. No capability, no access. Always,
   everywhere, no exceptions."** (design §6.1.1). An earlier draft of this plan
   proposed an `OwnershipMode { Unowned, Enforced }` enum branched on inside
   `ControlPlaneService` — i.e. an *exception* to default-deny, which is exactly
   what that sentence forbids. **Replaced** (§2.2): the unowned substrate is now
   expressed *inside* the model, by issuing a capability, not by skipping the
   check. There is no mode enum anywhere in B7.
2. **Wildcard allow, no wildcard deny** (design §6.1.1, ADR-0017 §2.1). Grants
   narrow by `*`-selector prefix cover and have **no exclusion operator** — "all
   except these five" is deliberately inexpressible in a grant. The policy layer
   owns exclusion (Zanzibar's `exclusion`, scoped inside a named permission) and
   the wildcard-deny analogue (`strict:` mode, ADR-0017 Open / design §9.7) —
   **both M04B, neither B7**. §3.1 states what B7b must therefore *not* build.
3. **Tier 1 is mis-addressed in the code, and ADR-0017 names B7 as its home**
   ("Reconcile in B7, which already touches this trust boundary"). This is new
   B7 scope that task.md predates entirely — see F3.

## 1. Flags — the reasoning record

Places where task.md is stale against the code, or where the design was
genuinely undecided. **All twelve are now resolved** — F3/F4/F8/F9/F12 by the
requester (§6), the rest in this plan's own design. Retained because they are
*why* the design is what it is, and because several document traps the obvious
implementation walks into: **F4's over-grant** (unowned ⇒ `substrate/admin`
would open DDL to everyone) and **F9's no-op** (`owner_did` on records the
substrate signs attributes nothing) are both mistakes that pass every other
test. Worst-first.

### F1 (resolved in design — §2.1) — `catalog.rs` is the wrong file; the deployed-app store is `EndpointRegistry`

task.md:368-370 and item 2 say:

> today `crates/app_orchestration/src/catalog.rs` records no owner/creator for
> a deployed app at all
> …
> **App catalog needs an owner field.** `catalog.rs` must record which DID's
> deploy call created each app.

Both sentences are true only in the trivial sense that `catalog.rs` records
*nothing* about deployed apps. Verified on this branch:

- `crates/app_orchestration/src/catalog.rs` is `ManifestCatalog` /
  `LocalFilesystemCatalog` — a **blueprint → `SynAppManifest` resolver** that
  reads TOML/JSON off disk. It has no deployed-app records, no state, no write
  path.
- The whole `syneroym-app-orchestration` deploy-side stack (`catalog.rs`,
  `journal.rs`'s `DeploymentJournal`, `reconcile.rs`'s `Reconciler`) runs
  **client-side, in `roymctl`**
  ([app.rs:82-109](apps/roymctl/src/commands/app.rs#L82)). The journal DB
  (`deployments.db`) lives on the *operator's* machine. `crates/control_plane`
  has a `Cargo.toml` dependency on the crate but **zero code references**
  (`grep -rn app_orchestration crates/control_plane/` → no hits; likely a dead
  dep worth removing separately).
- An owner field on `catalog.rs` would therefore record the owner *on the
  operator's laptop*, be unreachable from `ControlPlaneService::deploy`, and
  enforce nothing.

**The substrate's actual record of a deployed service is `EndpointRegistry`**
(`crates/core/src/local_registry.rs`), persisted through `EndpointStorage` →
`SqliteEndpointStorage` (`endpoints.db`,
`crates/data_db/src/registry_store.rs`). It is what
`ControlPlaneService::deploy` writes (`registry.register(...)`) and what `list`
reads (`registry.get_all_endpoints()`,
[service/orchestration.rs:404](crates/control_plane/src/service/orchestration.rs#L404)).

**Proposed resolution:** B7a's owner field goes on `EndpointRegistry` +
`EndpointStorage` (§2.1). task.md item 2 should be corrected to name it.
`app_orchestration` is not touched by B7 at all.

### F2 (resolved in design — §3.1; needs an ADR-0015 amendment) — `is_substrate_scope`'s wildcard defeats `substrate:` selectors

ADR-0015 A1 introduces selector paths on **both** resource forms
(`synapp:…/<selector>` *and* `substrate:<node_did>/<selector>`, with
`orchestrator`'s `app/<name>`) — then says:

> `Capability::covers` extends from `self.with == other.with` to a prefix
> cover; the `is_substrate_scope` wildcard rule is unchanged.

Those two clauses contradict each other. Today
([capability.rs:36](crates/ucan/src/capability.rs#L36)):

```rust
pub fn is_substrate_scope(&self) -> bool { self.0.starts_with("substrate:") }
```

and `grants` short-circuits on it:

```rust
if self.with.is_substrate_scope() { return self.can.entails(ability); }  // resource ignored
```

So B7b's intended deploy grant — `with: substrate:<node>/app/foo`,
`can: orchestrator/deploy` — starts with `substrate:`, hits the wildcard, and
**grants `orchestrator/deploy` on every app on the node**, `app/foo` silently
ignored. The selector is not merely unenforced; it is actively misleading.

**Proposed resolution:** the wildcard is the *unselectored* form. Narrow to:

```rust
/// A **bare** `substrate:<node_did>` — node-wide authority (ADR-0015 §1,
/// "`substrate/admin` ⊇ everything on that node"). A selector-bearing
/// `substrate:<node_did>/<selector>` (A1) is a specific resource and must
/// prefix-match like any other.
pub fn is_substrate_scope(&self) -> bool {
    self.0.starts_with("substrate:") && !self.has_selector()
}
```

This preserves both of A1's intents. **ADR-0015 A1's "unchanged" clause must be
amended** as part of B7b (same amend-in-place convention as ADR-0007/ADR-0011,
per the amendment's own preamble).

### F3 (resolved — §6 Q2; **new scope, from `f95206b`**) — Tier 1 is mis-addressed, and ADR-0017 assigns the fix to B7

ADR-0017's Open list (added in `f95206b`) and design §9.8 both say:

> `route_handler/dispatch.rs`'s `TODO(M04B/FDAE)` says which callers may reach
> a native service is enforced by FDAE "until then any verified identity
> passes." This ADR's position is that Tier 1 is a µs-scale capability check in
> the grant layer, not a policy-engine question — meaning that TODO belongs to
> the grant layer, not M04B, and the live gap (**today any verified identity
> reaches any native service**) is wider than the milestone docs imply.
> **Reconcile in B7, which already touches this boundary.**

task.md's B7 predates this entirely. The two mis-addressed sites:

- [dispatch.rs:84-89](crates/router/src/route_handler/dispatch.rs#L84) — the
  `TODO(M04B/FDAE)` above the `caller.cloned().ok_or_else(...)` anonymous gate.
- [io.rs:97-101](crates/router/src/route_handler/io.rs#L97) — the same TODO text
  on `build_caller`.

Note the anonymous half of that boundary is **already correct** and must not be
disturbed: design §6.1.2 confirms `caller = None` is a legitimate shipped state
— native interfaces reject it (that `ok_or_else`, the B0 gate), WASM guests
accept it (the guest arm never consults `caller`), which is why `AuthLevel` has
no `Anonymous` variant. B7 changes what a **verified** caller must hold, not
whether an anonymous one is admitted.

**Proposed resolution — split the TODO by tier, implement the half B7 already
owns:**

| Interface | Tier-1 check | Where |
|---|---|---|
| `orchestrator` | `orchestrator/*` names this node/app — deploy, undeploy, **and per-service `readyz`** (§2.4.1); the empty-`service_id` liveness ping stays open | **B7b** (§3.2) — it is the same gate B7b is already writing |
| `security` | `substrate/admin` | **Follow-on — carved out on review, see F3.1.** Q2 originally put it in B7b; that does not work. |
| `data-layer`, `vault`, `blob-store`, `messaging`, `app-config` | "does any capability name this service resource?" | **Follow-on**, recorded as a known gap |

#### F3.1 — why `security` cannot be gated in B7b (found on pre-implementation review, 2026-07-17)

The `security` interface's own TODO ([service.rs:118-122](crates/control_plane/src/service.rs#L118))
already names the right gate: *"security ops (KEK/secret) are node-owner
operations; final authorization is … against caller.session
(**substrate/admin**)"*. That is correct — and it is exactly the ability F4's
unowned posture **deliberately withholds** (issuing it would be the over-grant
trap: `substrate/admin` entails `data-layer/admin`, opening DDL to everyone).

Since every substrate is unowned in practice (F4 — nothing can create a
`ControllerAgreement`), gating `security` on `substrate/admin` means **nobody
can call `inject-kek`/`rotate-kek`/`set-secret` on any real deployment**,
including the legitimate operator, with no supported way to fix it. That is a
functional regression, not a tightening. The alternatives are worse:

- **Add `substrate/admin` to the unowned grant** — the over-grant trap. No.
- **Invent a distinct ability** (`substrate/security`-ish) purely so the unowned
  grant can carry it — a new vocabulary entry that exists only to route around
  the bootstrap posture, and one that has to argue KEK injection is *less*
  privileged than node admin. It is not: injecting a KEK unlocks every service
  DB on the node.

**Recommendation: B7b gates `orchestrator` only.** `security`'s gate ships with
the `ControllerAgreement` creation tool (§6.1), where it becomes coherent in one
step: ownership becomes establishable → `substrate/admin` becomes holdable →
the gate becomes meaningful → and *that* slice is the right place to reconsider
F4's default (fail-closed) anyway. Note the security interface is also absent
from task.md's B7 scope entirely — it arrived only via ADR-0017's Tier-1
reconciliation.

Blast radius if gated anyway, for the record: nothing in e2e or smoke touches
the `security` interface; the only affected test is
`test_security_dispatch_returns_sdk_statuses`
([service.rs:406](crates/control_plane/src/service.rs#L406)), which asserts the
*ungated* behavior and would have to change.

**Decided (§6 Q2): B7 does not close the five data interfaces**, and says so out
loud rather than leaving the TODO to imply M04B will. Reasons: (i) it is the
whole blast radius — every client of every deployed service would need a UCAN
naming that service, and none exists today (F5); (ii) design §6.1.2's intended
shape (*anonymous visitor → WASM guest → guest reads its own data as itself*)
means the interesting callers are guests, whose self-calls are identity-matched
in `ProxyRouter::check_native_capability_gate`
([proxy.rs:208](crates/router/src/proxy.rs#L208)), not capability-checked — so
Tier 1 there needs that gate's story reconciled too, which is its own design;
(iii) B7b's own gate proves the mechanism on a small surface first.

What B7 **must** do either way: retarget both TODOs from `M04B/FDAE` to the
grant layer with a named owner (`TODO(B7b)` / `TODO(post-B7)`), so the comment
stops asserting something the ADR now contradicts. That part is cheap and goes
in B7a.

### F4 (resolved — §6 Q1) — the unowned substrate, and "no exceptions"

`[iam].admin_ucan_root` is set in **no** config file and **no** e2e/smoke setup
— only in Rust tests (`grep -rn admin_ucan_root --include=*.toml --include=*.ts`
→ zero hits). No `ControllerAgreement` is created anywhere either: `roymctl` can
create identities and `DelegationCertificate`s but has **no command to produce a
`ControllerAgreement`**; `setup_substrate_identity` only *reads* one from
`config.identity.agreement`. So on every real deployment and every e2e run,
`admin_ucan_root` resolves to `None` and **no caller holds `substrate/admin`**.

Enforce naively and: `list` returns nothing to anyone; deploy denies everyone,
including `global-setup.ts:155`'s `roymctl svc deploy` — i.e. **the whole e2e
suite**. And there is no supported way to fix it, because nothing can mint the
agreement that would establish an owner.

**The tension `f95206b` creates.** Design §6.1.1 now says, absolutely:

> **The grant layer is default-deny.** No capability, no access. Always,
> everywhere, no exceptions.

An earlier draft of this plan proposed an `OwnershipMode { Unowned, Enforced }`
enum, branched on inside `ControlPlaneService` to *skip* the checks when no
owner exists. That is precisely an exception to "no exceptions", and it would
have scattered `if unowned` across three call sites.

**Proposed resolution — express the unowned state as a capability, not as an
exception.** An unowned substrate does not *skip* default-deny; it **issues
every verified caller the orchestrator abilities on this node**, and then
default-deny runs normally and passes. Same behavior, expressed inside the
model:

```rust
// crates/router/src/route_handler/io.rs, build_caller
match admin_root {
    // Owned: only the substrate owner, and they get substrate/admin (B0/B1
    // path, kept -- but the resource it names changes, see below).
    Some(root) if root == id.master_did => vec![Ability::SUBSTRATE_ADMIN],
    Some(_) => vec![],
    // UNOWNED (F4): no verified ControllerAgreement controller and no
    // [iam].admin_ucan_root -- nobody can root an orchestrator grant, so
    // default-deny would brick the substrate permanently (you could not
    // deploy the thing that would establish ownership). Every *verified*
    // caller therefore holds the orchestrator abilities here. This is not
    // an exception to default-deny (design §6.1.1): the capabilities are
    // really issued and every gate really checks for them. It is a
    // bootstrap posture -- Postgres's freshly-`initdb`'d cluster, not a
    // disabled check. Anonymous callers are unaffected: they are rejected
    // before this (design §6.1.2, dispatch.rs's B0 gate).
    //
    // NOT `substrate/admin`: see "the over-grant trap" below.
    None => vec![
        Ability::ORCHESTRATOR_DEPLOY,
        Ability::ORCHESTRATOR_UNDEPLOY,
        Ability::ORCHESTRATOR_STATUS,
    ],
}
// ...each pushed as:
Capability {
    // Bare `substrate:<node_did>` == node-wide (F2), and NOT
    // `substrate:<caller's master_did>` as B0 wrote it -- the resource is
    // the node, and F6's node check makes the difference load-bearing.
    with: ResourceUri::substrate(node_did),
    can: Ability(ability.to_string()),
    caveats: None,
}
```

**The over-grant trap — why not `substrate/admin` for the unowned case.** The
obvious version of this idea ("unowned ⇒ everyone is an admin") is a **security
regression**, and it is worth recording so nobody re-derives it. `Ability::entails`
([capability.rs:67](crates/ucan/src/capability.rs#L67)) opens with
`if self.0 == SUBSTRATE_ADMIN { return true }` — `substrate/admin` entails
*everything*, including `data-layer/admin`. So granting it to every verified
caller on an unowned substrate would open **`execute-ddl` and `query-raw`
(B0/B5's gates, `synsvc_native.rs:369-395`) to every verified caller** — strictly
worse than today, where `admin_root: None` means nobody holds it and DDL is
denied to all remote callers.

Issuing the three `orchestrator/*` abilities instead is exactly right, and the
existing `grants` short-circuit makes it work with no further change: a bare
`substrate:<node_did>` capability wildcards the *resource*, but `can.entails`
still has to hold, and `orchestrator/deploy` entails neither `data-layer/admin`
nor anything else. Verified against the B0/B5 gate:
`grants(synapp:…:svc:…, data-layer/admin)` → `is_substrate_scope()` true →
`orchestrator/deploy.entails(data-layer/admin)` → **false**. DDL stays denied;
today's posture is preserved exactly.

**Why this is worth the paragraphs:** it deletes the mode enum entirely.
`ControlPlaneService` never asks "am I owned?" — it asks "does this caller hold
node-wide orchestrator authority?", and the answer is *yes, for everyone* on an
unowned node. Every gate in §2.3/§2.4/§3.2 collapses to one predicate, the
unowned posture lives at exactly one site, and default-deny is literally true
everywhere.

It also states the security posture honestly: **an unowned substrate lets every
verified caller deploy, undeploy, and list every app.** That is what today's code
already does; B7 only makes it visible and bounds it to the orchestrator
surface. Boot warning (§2.2) says so in those words.

**Rejected alternative:** fail closed by default, with `[iam]
allow_unowned_deploy = true` opted into by e2e's generated `syneroym.toml`
(which `global-setup.ts:37-81` does write, so it is mechanically easy).
Rejected because nothing can create a `ControllerAgreement` today, so
`false`-by-default would brick every real deployment with no in-scope remedy,
and shipping a flag whose realistic value is always `true` is worse than naming
the posture. **Revisit if B7 also ships agreement creation** — then fail-closed
becomes the right default and this should flip.

### F5 (resolved in design — §2.5) — `roymctl` cannot act as a principal today

Every `roymctl` command builds `SyneroymClient::new(...)`
([svc.rs:67](apps/roymctl/src/commands/svc.rs#L67),
[app.rs:105](apps/roymctl/src/commands/app.rs#L105),
[security.rs:39](apps/roymctl/src/commands/security.rs#L39)), which **generates
a fresh ephemeral identity per invocation** (`generate_ephemeral_identity()`,
[sdk/src/lib.rs:160](crates/sdk/src/lib.rs#L160)), and never calls `with_ucan`.
So every deploy is attributed to a random DID seen once and never again;
`roymctl svc list` on an owned substrate returns nothing; and a re-deploy
arrives from a different DID than the original deploy (F7).

Attribution is close to useless without a stable operator identity, so **the
`roymctl --as` operator flag belongs in B7a, not B7b.** `roymctl identity create
--name X` already persists keys to `<dir>/identities/X.key`, and
`SyneroymClient::new_with_identity` already exists — this is wiring, not new
machinery (§2.5).

### F6 (resolved in design — §3.1) — `substrate:<other-node>` wildcard becomes exploitable at B7

Already documented and pinned by a test
(`substrate_scope_does_not_check_which_node_it_names`,
[capability.rs:345](crates/ucan/src/capability.rs#L345)), whose doc comment says
exactly this:

> once multi-node/owner-rooted trust exists (Slice B7) `covers`/`grants` should
> also verify the resource's node DID matches the node evaluating the
> capability.

Inert at B1 because the only issuer of a `substrate:`-scoped capability is this
node's own admin root. **Not inert at B7b**, which makes service owners
independent roots (A6): an owner-rooted chain carrying
`with: substrate:<any-string>` would wildcard everything on this node.

**Proposed resolution:** fold into B7b's A6 work — the resource-scoped predicate
is where the node check belongs, because it already has to distinguish "this
node's resources" from everything else:

```
is_trusted_root(issuer, resource) =
    (issuer == admin_root && resource_is_local(resource))
 || issuer == owner_of(resource)

resource_is_local(substrate:<did>[/…]) = (did == local_node_did)
resource_is_local(synapp:…)            = true   // evaluated locally by construction
```

Doing it there avoids threading `local_node_did` through
`Capability::grants`/`covers` (which would touch every call site and every
test). Update the pinning test to assert the new behavior. Note this covers
*chain-rooted* capabilities; the directly-pushed one in F4's `build_caller` is
covered by naming `substrate:<node_did>` there.

### F7 (resolved in design — §2.3; gap not in task.md) — deploy must reject ownership takeover

task.md item 2 says deploy records an owner, but never says what happens when
`deploy` targets a `service_id` that **already has a different owner**. Today
`deploy` on an existing id silently overwrites its endpoints. Letting it also
overwrite `owner_did` is a hostile-takeover path: any caller with deploy rights
re-deploys someone else's `service_id` and becomes its owner. Must fail closed
(§2.3). Flagged because it is a security requirement the recorded scope omits.

### F8 (resolved — §6 Q3; **sharpened by `f95206b`**) — `*` granularity: whole-segment only

ADR-0015 A1: "Matching is segment-wise prefix covering; a trailing `/` or `*` is
a prefix wildcard." Its examples are `blob/reports/` (trailing `/`) and `app/*`.
`f95206b`'s design §6.1.1 adds `collection/*` and
`synapp:acme:svc:shop/collection/*` — **all whole-segment**.

Undecided: is `app/acme-*` (a wildcard *within* a segment) legal? It is a
plausible want — "may deploy anything named `acme-…`" is how you would scope a
tenant — but no doc sanctions it.

**Decided (§6 Q3): whole-segment only** (`app/*`, `collection/*`,
`blob/reports/`). `covers_resource` treats a `*` segment as matching whole
segments, never a partial string. Reasons: "segment-wise prefix covering" is the
stated rule; every sanctioned example is whole-segment; and intra-segment
globbing is the classic prefix-confusion bug (`app/acme-` would also cover
`app/acme-evil-corp`, which reads like a typo, not a grant). If per-tenant name
prefixes are wanted later, that is an explicit ADR-0015 amendment, not an
inference. **No ADR change needed for this decision** — it is what A1 already
implies; the amendment §5 calls for is F2's, only.

### F9 (resolved — §6 Q4) — item 4 is **not implementable as written**; dropped from B7

*(Rewritten 2026-07-17 after review. An earlier draft of this flag proposed
adding an `owner_did` field to `EndpointInfo` and stamping it on "records the
substrate builds itself". That recommendation was **wrong on both halves** — see
"why the obvious fix is wrong" below. Recorded rather than deleted, because the
wrong version is the one you land on without reading the registry's verify
path.)*

**What is actually in the community registry — exactly two kinds of record:**

| Record | `service_id` | `endpoint_type` | `mechanisms` | Signed by | Created by |
|---|---|---|---|---|---|
| The substrate's own | `<node DID>` (== `substrate_id`) | `Substrate` | the Iroh addr/relay | the substrate's key | `build_signed_endpoint_info` ([runtime.rs:656](crates/substrate/src/runtime.rs#L656)), re-sent every heartbeat |
| Service → substrate, ×N | `<app DID>` | `Service` | **empty** — you reach it *via* the substrate | **the app's own key** | **`roymctl`, client-side** ([svc.rs:78-88](apps/roymctl/src/commands/svc.rs#L78)) |

The substrate only **relays** the second kind: `deploy` writes
`manifest.registry_certificate` to `hosted_apps_dir/<service_id>.json`, and the
heartbeat re-`register`s that pre-signed blob verbatim. Note both calls pass
`sync_dht: false` ([runtime.rs:609,641](crates/substrate/src/runtime.rs#L609)) —
**the heartbeat is HTTP-community-registry only, never DHT.**

**Why the substrate structurally cannot publish a `Service` record.** The
registry's contract is *a record for DID X must be signed by X*, enforced in
`SignedEndpointInfo::verify()`
([dht_registry.rs:101-117](crates/core/src/dht_registry.rs#L101)): it resolves
the expected pubkey **from `info.service_id`**, parses the packet against that
key, and hard-fails on `signed_packet.public_key() != expected_pkarr_pubkey`.
`verify_endpoint_signature` ([registry.rs:234](crates/community_registry/src/registry.rs#L234))
calls it on every `/register`. So a substrate-signed record naming app X is
**rejected** — and that is a sound property, not an oversight: it is what stops
a hostile substrate from publishing records for services it does not host.

**Why the obvious fix is wrong**, both halves of it:

- **"Stamp `owner_did` only on records the substrate builds itself"** stamps it
  on *the substrate's own record only* — whose owner is the substrate owner, not
  any service owner. It attributes nothing about any app. It is a no-op dressed
  as a feature.
- **A new `owner_did` field is redundant.** `EndpointInfo.delegation:
  Option<DelegationCertificate>` **already exists and is already verified**
  ([dht_registry.rs:138-145](crates/core/src/dht_registry.rs#L138)): it requires
  `cert.temporary_did == info.service_id` and checks `cert.verify(&cert.master_did)`.
  That is precisely "this service key acts for master DID `<owner>`" — the
  attribution item 4 asks for, already designed, already enforced, and
  **currently `None` at every one of the 18 construction sites.** Adding
  `owner_did` beside it would be a second, unverified spelling of the same
  claim — strictly worse, since `delegation` is *proof* and `owner_did` would be
  mere assertion.

**And item 4's *premise* is unsound — but so was this flag's first rebuttal.**
*(Two corrections, both 2026-07-17 on requester challenge. Draft 1 called
non-discoverability "the real gap" and proposed a `roymctl` force-publish fix.
Draft 2 over-corrected: it called non-discoverability "a supported posture" and
claimed "the model already spans the whole spectrum". **Both wrong.** The truth
is that publication today is neither a gap nor a designed posture — it is
**incidental**, and the feature that would make it deliberate does not exist.)*

Item 4 justifies itself as closing "a gap between 'owner believes it's deployed'
and 'it's actually discoverable'". Two things are true at once:

1. **A client *can* connect to an unpublished service** — so it is not simply
   broken. The owner shares the substrate address out of band and the client
   uses `SyneroymClient::new_with_mechanisms`, whose `provided_mechanisms` short
   -circuits the registry lookup entirely
   ([sdk/src/lib.rs:234-245](crates/sdk/src/lib.rs#L234)). The `Service` record
   carries `mechanisms: vec![]` and only maps service_id → substrate_id, so a
   client who already knows the substrate gains nothing from it anyway.
2. **But nothing *declares* that intent, and the gaps that follow are real.**
   Publication is currently a side effect of whether a pre-signed certificate
   happened to be handed in — an artifact, not a declaration.

**What does not exist** (verified, not assumed):

| | Status |
|---|---|
| **A manifest visibility flag** | **Absent at every layer.** `app_orchestration::models::ServiceConfig` ([models.rs:210](crates/app_orchestration/src/models.rs#L210)) has no visibility field; WIT `service-config` ([control-plane.wit:23](crates/wit_interfaces/wit/control-plane/control-plane.wit#L23)) has none. `deploy-manifest` carries only `registry-certificate: option<string>` — *an artifact you either supply or don't*, which is why the resulting posture is incidental. |
| **Sharing a service record privately** | **No such flow.** `SignedEndpointInfo` is a self-contained signed blob that *could* be handed to a peer, but nothing exports or imports one. The one out-of-band path (`new_with_mechanisms`) takes raw `mechanisms`, not a signed record — so it carries no signature, no nickname, no TTL, and no attribution. |
| **Cross-node service → service, unpublished target** | **Broken.** `ProxyRouter::invoke_remote` resolves *only* via the registry (`net_iroh::resolve_iroh_addr` → `registry_client.lookup`, [proxy.rs:322](crates/router/src/proxy.rs#L322), [net_iroh.rs:133](crates/router/src/net_iroh.rs#L133)). There is no provided-mechanisms escape hatch on this path, so **an unpublished service is unreachable from another node's services** — including sibling services of its own app. Same-node is fine (`registry.lookup` on the local `EndpointRegistry`). |
| **`is_private`** | Exists and is honored — but it gates **propagation to the parent registry** ([registry.rs:226](crates/community_registry/src/registry.rs#L226)), i.e. federation scope, *not* "publish or not". Hardcoded `false` at all 18 construction sites; no `roymctl` flag. Unreachable in practice. |

So the shape the requester describes — **the manifest declares whether the
record is public; a private service is still reachable by clients and by sibling
services that were given its record** — is a coherent feature that **does not
exist yet**, and is not what item 4 asks for either. Item 4 asks the *substrate*
to publish *more*; the actual need is to make publication *declared*, and to
make privacy not mean unreachable. Recorded as a follow-on (§6.2), not B7:
ownership is authorization on the substrate, discoverability is the registry,
and B7 needs neither of the above.

**What survives for B7's follow-on list is narrow and separate:** the
certificate must be signed by the key whose `did:key` **is** `--svc-id`
(`verify()` resolves the expected pubkey from `info.service_id`), so
`roymctl svc deploy --svc-id X --identity Y` where `Y`'s DID ≠ `X` silently
builds a certificate the registry rejects at `/register`, forever, while
printing success. A validation bug, unrelated to ownership.

**Decided (§6 Q4): item 4 is dropped from B7** and recorded as a
registry-trust-model question, not an ownership-attribution one. The `roymctl`
half — always emit the certificate, closing the real "deployed but
undiscoverable" gap — is **a separate PR** (§6.2). Making the substrate publish
for a service needs one of:

1. the substrate holding the service's key — no;
2. relaxing `verify()` to accept a substrate-signed record for X carrying a
   proof that X's owner authorized this substrate to host it — a real change to
   the registry's trust model, and its **own ADR**, not a bullet in B7; or
3. keeping the owner pre-signing and making `roymctl` always produce the cert —
   a client-side fix, independent of everything else in B7.

None of the three is ownership attribution, which is what B7 is about, and
option 2 is the only one that matches item 4's actual words. B7a's owner store
gives a future implementation of (2) the `owner_of` it would need — which is the
right relationship between the two pieces of work.

  `if !service_id.is_empty()`); it is a pure "is the node up?" probe. **This is
  what `SyneroymClient::wait_for_ready` calls, with `json!({})`**
  ([sdk/src/lib.rs:326](crates/sdk/src/lib.rs#L326)) — and *every* `roymctl`
  command and SDK client runs `wait_for_ready` during `connect()`
  (`svc.rs:68`, `security.rs:40/61`, the perf harness, tests). It runs before
  the caller has any capability. **Gating it — as both reviewers' "gate readyz
  like deploy" would — denies `wait_for_ready` to every ordinary client on an
  owned substrate, breaking connect for anyone reaching a *public* service.**
  Stays open, as a health probe (design §6.1.2's spirit: liveness is not an
  authorization surface).
- **Non-empty `service_id` — a per-service readiness check.** *This* is task.md
  item 1's "status-check": "is app X ready?" Gate it on `orchestrator/status`
  against `substrate:<node>/app/<service_id>`, exactly like `deploy` gates on
  `orchestrator/deploy` (§3.2).

```rust
async fn readyz(&self, service_id: String, caller: &CallerContext) -> Result<(), String> {
    if !service_id.is_empty() {
        // Per-service status-check (task.md item 1). Node-wide authority
        // (owner, or anyone on an unowned substrate -- F4) passes; otherwise
        // the caller needs orchestrator/status for this app.
        if !self.has_node_wide_orchestrator_authority(caller) {

            if !caller.has_capability(&resource, &Ability(Ability::ORCHESTRATOR_STATUS.to_string()))
            {
                return Err(format!("caller {} holds no orchestrator/status grant for \
                                    '{service_id}'", caller.caller_did));
            }
        }
        // ... existing container-readiness check, unchanged ...
    }
    // Empty service_id: substrate-liveness ping. Open -- wait_for_ready
    // depends on it, pre-capability.
    Ok(())
}
```

### 2.6 Multi-hop delegation — flag, don't misattribute (task.md item 3)

task.md is explicit that multi-hop delegation is **deferred, and must not
silently misattribute**:

> Multi-hop delegation (a delegate re-delegating further) is not resolvable with
> today's one-hop `DelegationCertificate` format and is explicitly deferred … —
> flag, don't silently misattribute.

One-hop resolution is free (F11). The *flagging* is not, and it is the half
that is easy to skip. `DelegationCertificate` has no proof chain, so a 2-hop
delegation is indistinguishable from a 1-hop one at the router — there is
nothing to detect at runtime. The honest implementation is therefore
documentation plus a test that pins the boundary:

- Doc comment on `EndpointRegistry::set_owner` and in B7a's `status.md` section:
  the recorded owner is the **`DelegationCertificate`'s `master_did`** — one
  hop, and by construction the only hop the cert format can express. A principal
  reached through two hops is recorded as the *intermediate* master, not the
  origin. Real chains are B7b/UCAN (`CapabilityToken.proofs`), where
  `anchor_did` (ADR-0015 A5) is the correct attribution terminal.
- Test: `owner_is_the_delegation_master_not_the_temporary_key` (§4).

### 3.1 `syneroym-ucan` — ADR-0015 A1/A3/A4 + F2/F6

**A1 — selectors** (`crates/ucan/src/capability.rs`):

```rust
impl ResourceUri {
    /// The `[/<selector>]` tail introduced by ADR-0015 A1, if any. The base is
    /// `synapp:<app>:svc:<svc>` or `substrate:<node_did>`; a selector is
    /// interface-shaped (`collection/<name>[/<id>]`, `blob/<prefix>`,
    /// `topic/<pattern>`, `rpc/<method>`, `app/<name>`).
    fn split_selector(&self) -> (&str, Option<&str>);
    fn has_selector(&self) -> bool;

    /// Segment-wise prefix cover (A1): bases must be equal, and `self`'s
    /// selector segments must be a prefix of `other`'s. No selector at all
    /// covers every selector on that base. A trailing `/` or `*` is an
    /// explicit prefix wildcard, matching whole segments only (F8).
    #[must_use]
    pub fn covers_resource(&self, other: &Self) -> bool;
}
```

Parsing note: `substrate:<node_did>` bases contain `:` inside the DID
(`did:key:z6Mk…`), so **parse the base by structure, not by splitting on `:`**.
`did:key` values are z-base-32 and contain no `/`, so splitting the *selector*
at the first `/` after the `substrate:`/`synapp:` prefix is sound. Unit-test
both forms explicitly.

**Wildcard semantics — allow only, per `f95206b`.** Three properties this must
have, and one it must *not*:

- `collection/*` / `app/*` covers every sibling — this is the mechanism design
  §6.1.1 relies on for "100 objects, 5 need rules… **nothing is enumerated**".
- `*` matches **whole segments only**, never a partial string (F8's
  recommendation — confirm before writing).
- No selector ⇒ covers every selector on that base (a service-granularity grant
  keeps meaning what it means today).
- **No exclusion operator. Do not build one.** Design §6.1.1 and ADR-0017 §2.1
  are explicit that a grant cannot express "all except these five", and that
  this is *deliberate*: the wildcard grant hits the five policies and the policy
  layer narrows. Exclusion lives in M04B (`exclusion`, scoped inside a named
  permission — design §6.2/§9.3), and its wildcard-deny cousin (`strict:` mode,
  design §9.7 / ADR-0017 Open) is an undecided M04B open. **If a "deny this app"
  need surfaces during B7b, it is a revoke, not a negative grant** (revocation
  is F11 — already built).

Then:

```rust
pub fn grants(&self, resource: &ResourceUri, ability: &Ability) -> bool {
    if self.with.is_substrate_scope() { return self.can.entails(ability); }  // bare form only (F2)
    self.with.covers_resource(resource) && self.can.entails(ability)
}
pub fn covers(&self, other: &Capability) -> bool {
    (self.with.is_substrate_scope() || self.with.covers_resource(&other.with))
        && self.can.entails(&other.can)
}
```

`is_substrate_scope` narrows per F2. Existing behavior is preserved for every
unselectored resource (`covers_resource` reduces to base equality), so B1's
tests should pass unchanged — **if any fails, that is a real regression, not a
test to update** (except the two pinning tests named in F2/F6, which change
deliberately).

**A2 — `orchestrator/*` entailment. Decided (§6 Q6): flat.** The three constants
already landed in B7a (§2.2); **no `Ability::tier` entry is added**, so each
entails only itself and they are granted together. This keeps `tier` a
`data-layer` special case, and keeps "deploy but not undeploy" expressible — a
tier would make it impossible. task.md item 1's "coarse ceiling — you deploy
here or you do not" (design §9.6) is still served: the coarse grant is all three
abilities at once, not one ability that implies the others.

**A3 — `can_delegate` caveat.** The minimum B7b needs (`where`/`fields` are
data-layer-only and belong to M04B, design §6.6):

```rust

    /// composition along a chain is logical AND.
    fn can_delegate(&self) -> bool {
        self.caveats.as_ref()
            .and_then(|c| c.get("can_delegate"))
            .and_then(serde_json::Value::as_bool)
            .unwrap_or(true)

```

Enforced in `token.rs`'s `granted_capabilities`, at the one place a parent backs

This is A4's "check the shape, stack the constraints" for the one caveat B7b
uses: `covers` checks `with`/`can`; `can_delegate` is *checked*, not conjoined,
because it is terminal rather than intersective. **A4's `where`/`fields`

([capability.rs:308](crates/ucan/src/capability.rs#L308)) must be **narrowed to

**A6 + F6 — resource-scoped `is_trusted_root`.** `ChainVerifyOpts`'s signature

### 3.2 The gate (task.md item 1 + F3's orchestrator half)

`ControlPlaneService::deploy`, after §2.3's takeover check:

```
// B7b: Tier-1 deploy admission. The caller must hold `orchestrator/deploy`
// covering this app. Note there is no owner/unowned branch and no separate
// substrate-owner bypass: a bare `substrate:<node>` capability (the owner's
// `substrate/admin`, or the unowned grant of F4) is `is_substrate_scope`, so
// `grants` wildcards the resource and only `entails` has to hold -- both
// pass here for free. An app-scoped B7b grantee is prefix-covered instead.
// One check, three principals, no branch.
let resource = ResourceUri(format!("substrate:{}/app/{service_id}", self.node_did));
if !caller.has_capability(&resource, &Ability(Ability::ORCHESTRATOR_DEPLOY.to_string())) {
    return Err(format!("caller {} holds no orchestrator/deploy grant for '{service_id}' \
                        on this substrate", caller.caller_did));
}
```

Same shape for `undeploy` (`ORCHESTRATOR_UNDEPLOY`) and for the **per-service**
form of `readyz` (`ORCHESTRATOR_STATUS`; the empty-`service_id` liveness ping
stays open — full rationale and code in §2.4.1). **Do not gate `list` on an
ability:** §2.4's owner filter already answers it, and an ungranted caller
correctly sees an empty list rather than an error.

Trace the three principals through it once, because this is the slice's core
and the wildcard interaction is easy to get wrong:

| Caller | Holds | `grants(substrate:<node>/app/foo, orchestrator/deploy)` |
|---|---|---|
| Unowned-substrate caller (F4) | `{substrate:<node>, orchestrator/deploy}` | bare ⇒ `is_substrate_scope` ⇒ `entails` ✓ |
| Substrate owner | `{substrate:<node>, substrate/admin}` | bare ⇒ `is_substrate_scope` ⇒ `entails` (admin ⊇ all) ✓ |
| App-scoped grantee | `{substrate:<node>/app/*, orchestrator/deploy}` | selector ⇒ prefix cover `app/*` ⊇ `app/foo` ✓ |
| Grantee, wrong app | `{substrate:<node>/app/bar, orchestrator/deploy}` | selector ⇒ `app/bar` ⊅ `app/foo` ✗ |
| Ordinary caller, owned substrate | nothing | ✗ |

Grant shape the substrate owner issues (`CapabilityToken::issue`, rooted at the
owner, audience = grantee):

```json
{ "with": "substrate:<node_did>/app/*",
  "can": "orchestrator/deploy",
  "caveats": { "can_delegate": false } }
```

(`app/*` = every app on the node — the coarse binary grant of design §9.6.
Narrower per-app grants name the app: `.../app/acme-shop`. Whether
`.../app/acme-*` is legal is F8.)

### 6.1 Consequences worth carrying forward

Two things follow from the decisions rather than being decided by them, and both
belong in `status.md` at B7 close:

1. **B7b's gate is inert until something can establish substrate ownership.**
   Q1 chose the bootstrap posture over "fail closed + ship a
   `ControllerAgreement` creation tool", and nothing else in the tree can create
   an agreement — so in practice every substrate is unowned, every verified
   caller holds the orchestrator abilities, and B7b's gate never denies. That is
   the correct sequencing (B7b builds the mechanism the tool will switch on),
   but B7 must not be reported as "deploy is authorized". **The agreement tool
   is the natural next slice after B7**, and it is the point at which F4's
   `allow_unowned_deploy` alternative should be reconsidered as the default.
2. **After B7, on an *owned* substrate, any verified identity still reaches any
   of the five data native-capability interfaces** (Q2). ADR-0017's Open list
   names this as the live gap; B7 narrows it to those five and retargets the
   TODO (§2.8), but does not close it.

### 6.2 Spun out of B7

- **Declared service visibility** (F9; requester 2026-07-17) — **now designed:
  [ADR-0018](../../../decisions/0018-service-record-visibility.md)** (status
  *Proposed*). It turned out much smaller than this entry assumed, because a
  `Service` record is only the mapping `service_id → substrate_id` (the
  addressing is grafted from the substrate's own public record at
  `lookup(_, resolve=true)`), and because same-node siblings already resolve via
  the local `EndpointRegistry`. Summary of what it decides: a three-valued
  `visibility` (`public`/`internal`/`private`, default `private`) on the manifest
  and WIT `service-config`; `SignedEndpointInfo` as the export/import artifact;
  a verified local known-records store consulted ahead of the registry; and
  `is_private` retained as the `internal` tier's wire encoding. The original
  sketch of the problem, retained:
  1. **The manifest declares it** — a visibility field on `ServiceConfig` (Rust
     *and* WIT `service-config`), so it is a property of the service, not of how
     it was deployed. Nothing exists at either layer today.
  2. **Private ≠ unreachable.** A private service's record should still be
     *shareable* — exported and handed to a client, or to a sibling service in
     the same app — so holders can connect while the world cannot discover it.
     No export/import flow exists; the only out-of-band path
     (`new_with_mechanisms`) takes raw mechanisms rather than a signed record.
  3. **Fix the cross-node hole this exposes.** `ProxyRouter::invoke_remote`
     resolves *only* through the registry, so an unpublished service is
     unreachable from another node — including from its own app's sibling
     services. Any visibility design must give that path a non-registry
     resolution route, or "private" silently means "same-node only".
  4. **Reconcile `is_private`**, which today means "don't propagate to the
     parent registry" — federation scope, not publication. Decide whether the
     new flag subsumes it or composes with it.

  Cross-cutting (manifest → WIT → orchestrator → proxy → SDK → `roymctl`).
  **Not B7**: ownership is authorization on the substrate; this is
  discoverability. One seam to keep in view — ADR-0018's known-records store and
  its per-service `visibility` row both live in `endpoints.db` alongside B7a's
  `service_owners`, so whichever lands second adds its tables to the same
  schema block rather than opening a parallel store.
- **`roymctl svc deploy` validates `--identity` against `--svc-id`** (Q4, scope
  corrected twice — see F9). The certificate must be signed by the key whose
  `did:key` **is** `--svc-id`, so `--svc-id X --identity Y` with `Y`'s DID ≠ `X`
  silently builds a certificate the registry rejects at `/register`, forever,
  while printing success. Fail at deploy time instead. Small, independent,
  and worth doing regardless of the visibility design above. Separate PR.
- **Relaxing the registry trust model** so a substrate may publish a `Service`
  record for an app it hosts, carrying a proof the app's owner authorized it
  (F9 option 2). This is what task.md item 4 literally asked for — but note F9
  also found its *motivation* unsound (non-discoverability is a posture, not a
  gap), so this needs a real consumer before it is worth doing. It changes
  `SignedEndpointInfo::verify`'s contract and needs its own ADR. B7a's owner
  store would give it the `owner_of` it needs.
- **A `ControllerAgreement` creation tool** (§6.1 item 1) — **and with it, the
  `security` interface's `substrate/admin` gate** (F3.1), which is unholdable
  until ownership is establishable. Reconsider F4's fail-closed default in the
  same slice: all three become coherent at once.
- **Multiple substrate owners** (Q5/F12).
- **Tier 1 for the five data native-capability interfaces** (§6.1 item 2).
