# Slice B5 — Privileged `query-raw` Escape Hatch — Implementation Plan

> **Requirement:** `[PLT-DAT]`; closes M04A gate item #3.
> **Depends on:** B0 (done — the `data-layer/admin` Admin-UCAN gate, the
> `CallerContext` identity threaded through native dispatch, and the
> `CallerContext::local_elevated` lifecycle identity all exist and are the
> exact mechanism reused here).
> **ADR:** [ADR-0011](../../../decisions/0011-privileged-raw-sql-query.md)
> (status *Proposed*), with **two decided deviations** recorded in §0.1 below —
> the ADR's own signature is self-contradictory against its stated use case, so
> it must be amended as part of this slice.

---

## 0. Summary of the change

Add a privileged `query-raw` capability to the `data-layer` `store` interface:
a caller holding `data-layer/admin` on the service resource can run arbitrary
**read-only** SQL against its own per-service encrypted SQLite DB, with values
always bound via `?` placeholders. It is the relief valve for expressivity the
safe JSON filter DSL (ADR-0007) will never grow (joins, aggregation, window
functions, CTEs). The gate is the **same Admin-capability check B0 already uses
for `execute-ddl`** — copied verbatim, not a new authorization concept.

Surfaces touched, in dependency order:

1. **WIT** (`data-layer.wit`, both copies) — add `sql-value` variant, a new
   `raw-query-result` record, and the `query-raw` function.
2. **`syneroym-data-db`** — `ServiceStore::query_raw` trait method + two impls;
   a `do_query_raw` reader-pool helper enforcing read-only.
3. **`syneroym-sandbox-wasm`** — guest-side `store::Host::query_raw` with the
   Admin gate (mirrors `execute_ddl` at `host_capabilities.rs:485`).
4. **`syneroym-control-plane`** — native-side `query-raw` arm in
   `dispatch_data_layer` with the Admin gate + a `SqlValueDto`/result DTO
   (mirrors the `execute-ddl` arm at `synsvc_native.rs:318`).
5. **Tests** — unit (data-db), guest-gate (sandbox_wasm), native-gate +
   injection (control_plane / router), per ADR-0011 §Consequences and
   task.md's Failure/Security table.

No config change (the Admin gate and `[iam].admin_ucan_root` already ship from
B0). No change to the writer loop, KEK/DEK, or DB-isolation model.

### 0.1 Decided deviations from ADR-0011 (must amend the ADR)

Both confirmed with the requester on 2026-07-15; the ADR is *Proposed*, so
amend it in this slice rather than implement a contradiction.

- **D1 — Return shape.** ADR-0011:39 writes
  `query-raw ... -> result<query-result, data-layer-error>`. `query-result` is
  `list<record-read-value>` — a **fixed** `{id, payload, creator-id,
  created-at, updated-at}` shape. That cannot represent an arbitrary
  projection/aggregation (`SELECT category, count(*) ...`), which is the ADR's
  own motivation and reference-scenario step 24 ("a report needing a join").
  **Decision:** return a new `raw-query-result { columns: list<string>, rows:
  list<row> }`, where `row = list<sql-value>`. Amend ADR-0011's signature block
  and add a short "Return shape" subsection recording why `query-result` was
  rejected.
- **D2 — Read-only enforced.** ADR-0011 gestures at "arbitrary DML/query SQL".
  **Decision:** `query-raw` accepts **read-only statements only**, enforced via
  `rusqlite::Statement::readonly()` (SQLite classifies the compiled statement —
  no SQL parser needed), executed on the concurrent **reader pool**. Writes keep
  using `put`/`patch`/`batch-mutate`; schema changes keep using `execute-ddl`
  (writer loop). A non-read statement returns `permission-denied`. Record this
  in the ADR's Decision + Consequences.

## 1. WIT surface (`syneroym:data-layer@0.1.0`, additive minor)

Two **byte-identical** copies must both change (verified identical today):
- `crates/wit_interfaces/wit/data-layer/data-layer.wit` — drives the guest
  `wit_bindgen::generate!` (`data_layer.rs`, world `data-layer-guest`), used by
  `test-components/`.
- `crates/wit_interfaces/wit/host/deps/data-layer/data-layer.wit` — drives the
  wasmtime `bindgen!` (`host.rs`, world `host-environment`), which is what
  produces both the `store::Host` trait (implemented in `sandbox_wasm`) **and**
  the `host_store` types re-exported by `data_db`
  (`crates/data_db/src/lib.rs:12`).

Add to the `store` interface in **both** files (place `sql-value` /
`raw-query-result` near `query-result` at line 39-42, and the function after
`query` at line 78):

```wit
/// A single bound parameter for `query-raw`. Values are always bound via `?`
/// placeholders -- never interpolated into the SQL text.
variant sql-value {
    text(string),
    integer(s64),
    real(f64),
    boolean(bool),
    null,
}

/// One result row: cells positionally aligned with `raw-query-result.columns`.
type raw-row = list<sql-value>;

/// Result of `query-raw`. Unlike `query-result`, this carries an arbitrary
/// projection (any columns a SELECT/CTE/aggregation produces), not the fixed
/// record shape.
record raw-query-result {
    columns: list<string>,
    rows: list<raw-row>,
}

/// Privileged raw-SQL escape hatch (ADR-0011). Gated by the `data-layer/admin`
/// capability (same trust boundary as `execute-ddl`); returns
/// `permission-denied` otherwise. Read-only: a statement that would mutate the
/// DB is rejected. `params` are bound via `?` placeholders in order.
query-raw: func(sql: string, params: list<sql-value>) -> result<raw-query-result, data-layer-error>;
```

> **WIT note.** `raw-row` as a named `type` alias is optional sugar; if
> `wit-bindgen`/`bindgen!` friction appears, inline `list<list<sql-value>>` for
> `rows` and drop the alias. Confirm both generators accept a `type` alias
> before relying on it (the repo currently uses none). **Default to inlining**
> `rows: list<list<sql-value>>` to avoid the risk.

**Generated-type consequences** (no manual conversion layer — host crates speak
generated types directly, per AGENTS.md):
- `host_store` gains `SqlValue` (enum) and `RawQueryResult` (struct) with the
  `additional_derives` already configured (`serde::Serialize/Deserialize`).
- The `store::Host` trait gains a required
  `async fn query_raw(&mut self, sql: String, params: Vec<SqlValue>) ->
  Result<RawQueryResult, DataLayerError>` — `sandbox_wasm` must implement it or
  it won't compile.
- The guest `data-layer-guest` bindings gain a `query-raw` import — additive,
  existing guests unaffected; `wasm32-wasip2` stays buildable.

## 2. `syneroym-data-db`

### 2.1 `ServiceStore` trait (`crates/data_db/src/traits.rs`)

Add after `query` (after line 151), signature mirroring `execute_ddl`'s doc
tone (host trusts its caller re: authorization; this method only enforces
read-only + binding):

```rust
/// Executes a privileged read-only raw-SQL query (ADR-0011). Callers must
/// have already verified the `data-layer/admin` capability -- this method
/// trusts its caller for authorization but enforces two invariants itself:
/// (1) `params` are bound via `?` placeholders, never interpolated; (2) the
/// statement is read-only (a mutating statement returns `permission-denied`).
async fn query_raw(
    &self,
    sql: &str,
    params: &[host_store::SqlValue],
) -> Result<host_store::RawQueryResult, host_store::DataLayerError>;
```

### 2.2 Reader-pool helper (`crates/data_db/src/sqlite.rs`)

Add a free function next to `do_query` (~line 372). `SqlValue` (the rusqlite
type) is already imported as `types::Value as SqlValue` at line 16 — **name
clash** with the WIT `host_store::SqlValue`. Reference the WIT one fully as
`host_store::SqlValue` inside this function; keep `SqlValue` meaning the
rusqlite type (consistent with the rest of the file).

```rust
fn do_query_raw(
    conn: &Connection,
    sql: &str,
    params: &[host_store::SqlValue],
) -> Result<host_store::RawQueryResult, host_store::DataLayerError> {
    // Bind guest values positionally; never interpolate into `sql`.
    let bound: Vec<SqlValue> = params.iter().map(wit_to_rusqlite_value).collect();

    let mut stmt = conn.prepare(sql).map_err(|e| {
        // A malformed/privileged statement is a caller error, not internal.
        host_store::DataLayerError::SchemaViolation(format!("query-raw prepare failed: {e}"))
    })?;

    // Read-only enforcement (D2): SQLite classifies the compiled statement.
    // `readonly()` is false for INSERT/UPDATE/DELETE/DDL and PRAGMA writes.
    if !stmt.readonly() {
        return Err(host_store::DataLayerError::PermissionDenied);
    }

    let column_count = stmt.column_count();
    let columns: Vec<String> = stmt
        .column_names()
        .into_iter()
        .map(str::to_string)
        .collect();

    // Enforce the same page ceiling `query` uses, to bound memory.
    let mut rows_out: Vec<Vec<host_store::SqlValue>> = Vec::new();
    let mut rows = stmt
        .query(rusqlite::params_from_iter(bound.iter()))
        .map_err(map_rusqlite_error)?;
    while let Some(row) = rows.next().map_err(map_rusqlite_error)? {
        if rows_out.len() as u32 >= MAX_QUERY_PAGE_SIZE {
            break; // hard cap; documented, mirrors query()'s MAX_QUERY_PAGE_SIZE
        }
        let mut cells = Vec::with_capacity(column_count);
        for i in 0..column_count {
            cells.push(rusqlite_to_wit_value(row.get_ref(i).map_err(map_rusqlite_error)?)?);
        }
        rows_out.push(cells);
    }

    Ok(host_store::RawQueryResult { columns, rows: rows_out })
}
```

Two pure converters (same file, private):

```rust
fn wit_to_rusqlite_value(v: &host_store::SqlValue) -> SqlValue {
    match v {
        host_store::SqlValue::Text(s) => SqlValue::Text(s.clone()),
        host_store::SqlValue::Integer(i) => SqlValue::Integer(*i),
        host_store::SqlValue::Real(f) => SqlValue::Real(*f),
        host_store::SqlValue::Boolean(b) => SqlValue::Integer(i64::from(*b)),
        host_store::SqlValue::Null => SqlValue::Null,
    }
}

fn rusqlite_to_wit_value(
    v: rusqlite::types::ValueRef<'_>,
) -> Result<host_store::SqlValue, host_store::DataLayerError> {
    use rusqlite::types::ValueRef;
    Ok(match v {
        ValueRef::Null => host_store::SqlValue::Null,
        ValueRef::Integer(i) => host_store::SqlValue::Integer(i),
        ValueRef::Real(f) => host_store::SqlValue::Real(f),
        ValueRef::Text(bytes) => host_store::SqlValue::Text(
            String::from_utf8(bytes.to_vec()).map_err(|_| {
                host_store::DataLayerError::SchemaViolation(
                    "query-raw returned non-UTF-8 text".into(),
                )
            })?,
        ),
        // WIT `sql-value` has no blob arm (ADR-0011). Surface, don't corrupt.
        ValueRef::Blob(_) => {
            return Err(host_store::DataLayerError::SchemaViolation(
                "query-raw: BLOB columns are not representable in sql-value; \
                 project them via hex()/base64 instead"
                    .into(),
            ));
        }
    })
}
```

> **Design note (blobs).** `sql-value` intentionally has no `blob` arm (matches
> ADR-0011). Returning a typed `schema-violation` is the honest behavior; adding
> a `blob(list<u8>)` arm is a possible future WIT extension but is out of scope
> here — flag F2.

> **Boolean asymmetry.** `boolean(bool)` binds as integer 0/1 but never
> *returns* (SQLite has no boolean storage class, so results surface as
> `integer`). This is inherent and documented, not a bug — the input arm exists
> for ergonomic parameter binding. Flag F3.

### 2.3 Two `ServiceStore` impls (`crates/data_db/src/sqlite.rs`)

- `impl ServiceStore for SqliteServiceStore` (~line 1027): add `query_raw`
  following `query`'s reader-pool pattern (lines 1122-1137) — grab a
  `reader_pool` connection and `conn.interact(move |conn| do_query_raw(conn,
  &sql, &params))`. Clone `sql: String` and `params: Vec<host_store::SqlValue>`
  into the closure.
- `impl ServiceStore for Arc<SqliteServiceStore>` (~line 1180): forward
  `self.as_ref().query_raw(sql, params).await` (mirror line 1230-1236).

> **Reader-pool safety (D2).** `do_query_raw` rejects any non-read statement
> *before stepping* (`stmt.readonly()` is known post-`prepare`, pre-`query`), so
> the read-write-capable reader connection can never actually mutate — the
> single-writer invariant holds.

## 3. `syneroym-sandbox-wasm` — guest-side host impl

`crates/sandbox_wasm/src/host_capabilities.rs`, in `impl store::Host for
HostState` (add after `execute_ddl`, ~line 502). This is a near-verbatim copy
of the `execute_ddl` gate (lines 485-502):

```rust
async fn query_raw(
    &mut self,
    sql: String,
    params: Vec<SqlValue>,
) -> Result<RawQueryResult, DataLayerError> {
    // Admin-capability gate (ADR-0015/0016), identical to execute_ddl: only a
    // caller holding `data-layer/admin` on this component's own resource may
    // run raw SQL. Lifecycle init/migrate runs as `AuthLevel::LocalElevated`,
    // which carries it.
    let resource = ResourceUri::service(&self.component_id, &self.component_id);
    if !self.caller.has_capability(&resource, &Ability(Ability::DATA_LAYER_ADMIN.to_string())) {
        return Err(DataLayerError::PermissionDenied);
    }
    let store = open_store(
        self.component_id.clone(),
        self.key_store.clone(),
        self.storage_provider.clone(),
    )
    .await?;
    store.query_raw(&sql, &params).await
}
```

Import additions to the `use syneroym_wit_interfaces::host::syneroym::{...
store::{self, ...}}` group (lines 31-40): add `RawQueryResult, SqlValue` to the
`store::{...}` list.

## 4. `syneroym-control-plane` — native-side dispatch

`crates/control_plane/src/synsvc_native.rs`, in `dispatch_data_layer` (add a
new arm before the `other =>` fallthrough at line 343). Mirrors the
`execute-ddl` arm (lines 318-342) for the gate; adds a `SqlValueDto` because the
bindgen `SqlValue` variant serializes with serde's default externally-tagged
PascalCase (`{"Text": "..."}`), not this API's snake-case JSON convention —
same reason `MutationDto`/`IndexDefinitionDto` exist (lines 172-190, 287-296).

```rust
"query-raw" | "query_raw" => {
    // Admin-capability gate -- identical to `execute-ddl` above.
    let resource = ResourceUri::service(
        invocation.caller.app_instance.as_deref().unwrap_or(&self.service_id),
        &self.service_id,
    );
    if !invocation
        .caller
        .has_capability(&resource, &Ability(Ability::DATA_LAYER_ADMIN.to_string()))
    {
        return Err(data_layer_error(DataLayerError::PermissionDenied));
    }

    // Hand-rolled DTO: the bindgen `SqlValue` variant derives serde's default
    // PascalCase externally-tagged form; this API is snake_case tagged JSON.
    #[derive(serde::Deserialize)]
    #[serde(tag = "type", content = "value", rename_all = "snake_case")]
    enum SqlValueDto {
        Text(String),
        Integer(i64),
        Real(f64),
        Boolean(bool),
        Null,
    }
    #[derive(serde::Deserialize)]
    struct Req {
        sql: String,
        #[serde(default)]
        params: Vec<SqlValueDto>,
    }
    let req: Req = parse_params(&invocation)?;
    let params: Vec<SqlValue> = req
        .params
        .into_iter()
        .map(|p| match p {
            SqlValueDto::Text(s) => SqlValue::Text(s),
            SqlValueDto::Integer(i) => SqlValue::Integer(i),
            SqlValueDto::Real(f) => SqlValue::Real(f),
            SqlValueDto::Boolean(b) => SqlValue::Boolean(b),
            SqlValueDto::Null => SqlValue::Null,
        })
        .collect();
    let result = store.query_raw(&req.sql, &params).await.map_err(data_layer_error)?;
    to_payload(&result)
}
```

> **`Null` variant JSON.** With `#[serde(tag="type", content="value")]`, a unit
> variant `Null` is `{"type":"null"}` (no `value`). Confirm serde accepts the
> mixed unit/newtype variants under internally-adjacent tagging; if it rejects
> the unit variant, model `Null` as `Null {}` or switch the whole DTO to
> `#[serde(tag="type")]` (adjacent → internal) with `value` folded in. **Write a
> round-trip unit test for the `null` case specifically** (Flag F1).

**Imports** (`synsvc_native.rs:32`): add `SqlValue` to the
`use syneroym_rpc::{...}` list (re-exported there per B0 — verify; §6). The
result serializes fine via the existing `to_payload` because `RawQueryResult`
carries `serde::Serialize` from `additional_derives`; the *output* variant
tagging is PascalCase, which is acceptable for a response payload (no consumer
DTO expects snake_case on the way out) — **but** if the guest-to-guest proxy
round-trips this result back into a typed WIT value (A1's `wasm_results_to_json`
path), confirm the output encoding matches what the conversion layer expects.
Flag F4.

`data_layer_error` already maps `PermissionDenied → -32010` (line 100-104) and
`SchemaViolation → -32012`, so no error-mapping change is needed.

## 7. Flags — ambiguity / staleness / risk (resolve while coding)

- **F0 — ADR-0011 is self-contradictory and must be amended** (§0.1). It is
  *Proposed*, not *Accepted*; update the signature (D1), add read-only wording
  (D2), and bump status if the milestone process wants it *Accepted* on ship.
  Do **not** leave the ADR reading `-> query-result` while the code returns
  `raw-query-result`.
- **F1 — serde `null` DTO variant** (§4). Unit-test the `{"type":"null"}` round
  trip; adjust tagging if serde rejects the mixed unit/newtype enum.
- **F2 — BLOB columns** unrepresentable in `sql-value` (§2.2). Chosen behavior:
  typed `schema-violation`. Alternative future: add a `blob` arm — out of scope.
- **F3 — boolean is input-only** (§2.2): binds 0/1, returns as `integer`.
  Inherent to SQLite; document in the WIT doc-comment and ADR.
- **F4 — result encoding across the A1 proxy** (§4). If a `query-raw` result is
  ever proxied guest→guest and re-typed through A1's `wasm_results_to_json` /
  conversions, verify the PascalCase output variant tagging is what that path
  expects. Same-node native/guest calls are unaffected. (Likely a no-op for B5,
  but note it so A1's conversion assumptions aren't silently violated.)
- **F5 — MAX page cap on raw results** (§2.2). `do_query_raw` caps at
  `MAX_QUERY_PAGE_SIZE` (1000) with a hard `break`, matching `query`. Unlike
  `query` there is **no cursor** for raw SQL (arbitrary ordering) — a truncated
  result is silently capped. Decide: (a) silent cap (documented), or (b) return
  a `schema-violation`/`quota-exceeded` when a raw query exceeds the cap so the
  caller must add `LIMIT`. **Recommend (b)** — `quota-exceeded` on overflow is
  less surprising than silent truncation for a report query. Confirm before
  coding; the §2.2 pseudo-code currently shows (a).
- **F6 — task.md:438 wording.** The Failure/Security table row says
  `query-raw` without Admin returns "`data-layer-error::permission-denied`, same
  shape as `execute-ddl` today" — consistent with this plan. No staleness; noted
  for traceability.
