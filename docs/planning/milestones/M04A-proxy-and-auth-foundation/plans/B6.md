## 0. TL;DR of the recommendation

Adopt the **derived-KEK model (A)**: keep one substrate-global *master* KEK
injected once at startup (exactly as today), and change what wraps each
service's DEK from *the raw master KEK* to a **per-instance KEK derived via
HKDF-SHA256(master, info = "syneroym:kek:v1:{service_id}")**. The per-instance
scope is the **`service_id`** the DEK is already keyed by — which `io.rs:103`
documents as equal to the app-instance id today — so **no `StorageProvider` /
`ServiceStore` / dispatch signature changes are needed**; only `KeyStore`
internals change plus a tested re-wrap path.

This is the only variant buildable now (there is no per-instance KEK
*provisioning* channel in the codebase), it matches the slice body's "derive
per-app-instance KEKs", and it makes the "rotate under old global KEK →
per-instance KEK" re-wrap path (Migration Strategy) directly testable.

**It diverges from three literal phrasings in `task.md` that describe a
different model (B).** Those are called out in §6 (Flags) and must be
confirmed/reconciled before coding — this is the one load-bearing decision in
the slice.

## 1. Current state (re-verified on `main` @ `b2d32fa`)

Envelope encryption today (`crates/data_keystore/src/key_store.rs`):

- One **substrate-global** KEK held in `Arc<Mutex<Option<Zeroizing<[u8;32]>>>>`,
  injected once via `inject_kek(kek_bytes, _scope: Option<&str>)`. **`_scope` is
  dead** — every call site passes `None` (`service.rs:185` and all
  tests/benches).
- Per-service **DEK**, 32 random bytes, wrapped with **AES-256-GCM under the raw
  master KEK**, stored one row per `service_id` in `dek_store(service_id,
  encrypted_dek, nonce, created_at)` inside `substrate.db`.
- `generate_dek(service_id, conn)` / `load_dek(service_id, conn)` wrap/unwrap
  under the raw master KEK. `rotate_kek(new_kek, conn)` re-wraps every row from
  old→new raw master in one transaction.

Consumption chain (all keyed by `service_id`, none of it needs to change):

- `SqliteStorageProvider::resolve_dek` (`sqlite.rs:711`) → `load_dek` /
  `generate_dek`; the DEK is the SQLCipher `PRAGMA key` for the service DB
  (`open_service_db`, `sqlite.rs:940-946,978`).
- `load_service_dek` (`sqlite.rs:1015`) → same `resolve_dek`; used by blob
  content encryption (`host_capabilities.rs:632`, `synsvc_native.rs:198`) and
  the HTTP signed-URL path (`http.rs:916`). Blob/vault payloads are encrypted
  under the **DEK** (and, for blobs, an HKDF subkey of the DEK —
  `data_blob/src/crypto.rs`), **not** the KEK — so narrowing the KEK re-wraps
  DEKs only and touches no stored ciphertext (matches "No stored-data schema
  change").
- One `SynSvcNativeService` is registered per `service_id` at deploy
  (`orchestration.rs:380`) and dispatch resolves it by the preamble's
  `service_id` (`dispatch.rs:81`); it only ever opens `self.service_id`'s DB
  (`open_store`, `synsvc_native.rs:191-196`). WASM guests open only their own
  `HostState.component_id` DB. **A caller therefore cannot name another
  instance's DB** — cross-instance retrieval is structurally impossible, not
  merely gated (relevant to the failure test, §4.3).

Precedent for the crypto: HKDF-SHA256 derivation from key material already
exists in `crates/data_blob/src/crypto.rs` (`Hkdf::<Sha256>` with an `info`
string); `hkdf = "0.12"` and `sha2 = "0.10"` are workspace deps.

## 2. Design decision: derive (A) vs. provision-and-inject (B)

| | **(A) Derive from master — RECOMMENDED** | **(B) Inject a distinct KEK per instance** |
|---|---|---|
| KEK store | one master, injected once (unchanged) | `HashMap<scope, KEK>`; `inject_kek(kek, Some(scope))` per instance |
| Effective wrap key | `HKDF(master, service_id)` | the injected per-instance KEK |
| Provisioning needed | none | an authenticated per-instance inject channel (does **not** exist) |
| Independent unlock | ✗ (master derives all) — defense-in-depth only | ✓ (an un-injected instance stays locked) |
| Matches slice body | ✓ "derive per-app-instance KEKs" | ✗ |
| Matches Migration Strategy / exit-criterion wording | ✗ ("scope passed to `inject_kek`", "`_scope` actually used") | ✓ |
| Buildable in M04A | ✓ | ✗ (no consumer/UX for per-instance provisioning; consistent with A.4/A.6 "no consumer, defer") |

**Recommendation: (A).** (B) is the architecture's *eventual* "Unlock Model"
(`system-architecture.md:1808` — "the service remains locked upon node restart
until the owner provisions the KEK"), but it needs a per-instance provisioning
mechanism that has no consumer or management-channel surface in M04A. (A)
delivers the API-shape narrowing + forward-compatible derivation now, as
defense-in-depth, and leaves the door open for (B) to slot the *injected* key in
front of the derivation later. This mirrors how this milestone deferred every
other consumer-less item (A.4 streams, A.6 credits).

**Consequence for `CallerContext`:** under (A) the derivation scope is the
`service_id` the store is already bound to, so ADR-0016's `caller.app_instance`
is **not** consulted for KEK selection (it is always `None` today anyway, and
`io.rs:104` records `app_instance_id == service_id`). No new gate is added on the
data path; see §4.3 for why the failure test still holds structurally. **This
makes the B0 docstring on `CallerContext.app_instance`
(`crates/rpc/src/native.rs:25-27`: "App-instance the caller acts as
(`creator_id`, per-app KEK)") mildly misleading** — it implies the field drives
per-app KEK selection, which (A) does not do. Clarify it (§3.4), don't delete
the concept: the field still *names* the per-app identity; B6 just derives the
KEK from `service_id` (== that identity today) rather than reading this field.

The rest of this plan assumes **(A)**. §6-F1 states exactly what changes if the
answer is (B) instead.

### 2.1 Threat-model limitation of Model A (READ THIS — it is what B6 does NOT buy)

Model A derives every service's KEK from **one** injected master. Therefore
**anyone who can inject the master KEK, or read the substrate process RAM, can
derive every service's per-instance KEK and decrypt every service DB at rest.**
Model A buys only: a *leaked derived* `K_s` does not reveal the master or any
sibling `K_s'`. It does **NOT** buy at-rest isolation between services/tenants
against a substrate-level adversary or against whoever holds the master.

That stronger property — a party may unlock *their* service's DB without being
able to decrypt anyone else's — requires **independent, IAM-gated per-instance
KEK provisioning (Model B)**, where each service's KEK is a *separately injected*
secret, not a derivation of a shared master. **This is exactly the property
ADR-0006 scoped for M4** ("enforce which authenticated caller is authorised to
inject which app's KEK") and gated multi-tenant production behind ("M4 must
introduce per-SynApp-Instance KEK before any production multi-tenant deployment
is considered secure"). **B6/Model A does not satisfy that requirement** — it is
a forward-compatible down-payment; Model B is deferred (§10), *not dropped*, and
the multi-tenant-not-secure-until-B gate stays in force.

## 3. Exact changes (model A)

### 3.1 `crates/data_keystore/Cargo.toml`
Add:
```toml
hkdf.workspace = true
sha2.workspace = true
```

### 3.2 `crates/data_keystore/src/key_store.rs`

**New private helper** (mirrors `data_blob/src/crypto.rs`'s HKDF pattern):
```rust
// info string is domain-separated from data_blob's "syneroym:blob*" keys.
fn derive_instance_kek(master: &[u8; 32], scope: &str) -> Zeroizing<[u8; 32]> {
    let hk = Hkdf::<Sha256>::new(None, master);      // no salt; master is high-entropy
    let info = format!("syneroym:kek:v1:{scope}");
    let mut okm = Zeroizing::new([0u8; 32]);
    // `expect_used` is workspace warn-level; the precedent in
    // data_blob/src/crypto.rs pairs this exact expect with the same allow.
    #[allow(clippy::expect_used)]
    hk.expand(info.as_bytes(), okm.as_mut_slice())
        .expect("32 bytes is a valid HKDF-SHA256 output length");
    okm
}
```

**`inject_kek` — drop the dead `_scope` param** (honest "change in place", no
shim; per the "no migrations pre-release" convention):
```rust
pub fn inject_kek(&self, master_kek: [u8; 32]) -> Result<()>   // was (kek_bytes, _scope)
```
Body unchanged except the removed param. (FLAG §6-F2: this contradicts the
literal "`_scope` actually used" exit-criterion; reconcile there.)

**`generate_dek(service_id, conn)`** — unchanged signature; the only change is
the wrap key:
```rust
let master = self.get_kek()?;
let kek = derive_instance_kek(&master, service_id);   // <-- was: use master directly
let key = Key::<Aes256Gcm>::from_slice(&*kek);
// ...rest identical (random DEK, GCM encrypt, INSERT OR REPLACE)...
```

**`load_dek(service_id, conn)`** — unchanged signature; unwrap key changes the
same way:
```rust
let master = self.get_kek()?;
let kek = derive_instance_kek(&master, service_id);   // <-- was: use master directly
```

**`rotate_kek(new_kek, conn)`** — re-wrap per row using derived keys on both
sides:
```rust
// inside the row loop, decrypt with derive_instance_kek(&old_master, &service_id)
// and re-encrypt with derive_instance_kek(&new_master, &service_id).
```
(The row already yields `service_id`, so this is a two-line change to which key
each side uses.)

**No dedicated migration method** (decision at top of file). There is no
deployed M3-era `substrate.db` to upgrade, so a one-shot
raw-master→per-instance re-wrap is not built. The *re-wrap mechanism itself*
(unwrap under one derived key, re-wrap under another) is exercised and tested by
`rotate_kek` (§5, test 3) — that is what satisfies the Migration Strategy's
"test the re-wrap path" requirement.

**Update the doc comment** on the struct/`inject_kek`/`load_dek` to state the
per-instance derivation and the `service_id`-as-scope convention.

## 4. Behavioural specification & the failure test

### 4.1 Fresh deployments
New services generate a DEK wrapped under `HKDF(master, service_id)`
transparently — no config, no new op. `SubstrateConfig` **unchanged** (the
`[iam].admin_ucan_root` from B0 is unrelated; no new key here).

### 4.2 Existing (M3-era) DBs
A `substrate.db` written before B6 has raw-master-wrapped DEKs; after B6,
`load_dek` derives a *different* key and decryption fails. **Decided (user):
nothing is deployed, so dev state is simply wiped** — no migration, no
auto-detection, no scheme marker. Not a concern.

### 4.3 Failure-table row — "Per-app-instance KEK requested by a caller without
that app-instance's identity → permission-denied; another instance's KEK never
returned."

Under (A) this is realized **structurally + cryptographically**, not by a new
runtime gate:
- **Never returned across instances:** each `SynSvcNativeService`/`HostState` is
  bound to a fixed `service_id`/`component_id` and only ever calls
  `load_dek(self_id)`; a caller cannot address another instance's DB (§1). B0's
  handshake already rejects anonymous callers before native dispatch.
- **Cryptographic isolation:** a DEK wrapped for scope A cannot be unwrapped
  with the key derived for scope B (distinct HKDF `info`). This is the
  assertable core of the failure test (§5, `cross_instance_kek_isolation`).

Do **not** invent a new per-request "does caller own this instance" gate here —
that is the Tier-1 "may this caller touch this service at all" check explicitly
left open for the five data interfaces (`io.rs:147-156` TODO; B7b did it for
`orchestrator` only). Folding it in would exceed B6's scope. (FLAG §6-F4 if the
reviewer reads the failure row as demanding an explicit gate.)

## 10. Model B — deferred, NOT dropped (a security gate, tracked)

**Model B** = the architecture's "Unlock Model" (`system-architecture.md:1808`)
**and ADR-0006's literal M4 requirement**: a *distinct, externally-provisioned*
KEK per SynApp-instance (and, as the eventual target, per-service), IAM-gated on
*who may inject which app's KEK*, so an instance whose KEK has not been injected
stays locked at rest. This is the property that actually delivers **at-rest
isolation between tenants** — a party can unlock *their* DB without gaining the
ability to decrypt anyone else's, and neither substrate-RAM access nor holding
one service's KEK unlocks the rest. **Model A (B6) does NOT provide this** (see
§2.1): its single injected master derives every KEK, so substrate/master access
decrypts everything. Model A is a forward-compatible down-payment — the injected
key that front-runs the HKDF derivation simply becomes per-instance instead of a
single master — **not** a substitute.

**Security consequence — the multi-tenant gate stays shut.** ADR-0006 states "M4
must introduce per-SynApp-Instance KEK before any production multi-tenant
deployment is considered secure." Because B6 ships Model A, **that gate is NOT
cleared by M04A** — multi-tenant-at-rest security remains blocked on Model B.
This must be recorded as an open security item, not a soft "eventual target".

**Why deferred:** Model B needs an authenticated **per-instance KEK
provisioning channel** and its operator UX — deciding *who* may inject *which*
instance's KEK (ADR-0006's "enforce which authenticated caller is authorised to
inject which app's KEK"), likely realized as a UCAN capability (B1's chain) on
the `security` interface. There is no consumer, management surface, or ADR for
that yet in M04A — the same "no consumer, defer" reasoning this milestone
applied to A.4 (streams) and A.6 (credits).

**When / where it lands — deliberately NOT pinned to a milestone.** A likely
home is an M5 security-hardening / management-channel slice, but M5 is not
scoped yet and hard-committing it here would be a guess. It is gated on a real
provisioning consumer and should get its own ADR (KEK provisioning +
authorization model). Prerequisites already in place: B1 UCAN chains,
`CallerContext.session` capability checks, and this slice's derivation seam.
