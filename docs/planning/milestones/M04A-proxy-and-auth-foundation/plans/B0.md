# Slice B0 ‚Äî Native-Dispatch Authentication Gap Closure ‚Äî Implementation Plan

> Requirement `[FND-IAM]` foundation; closes M3‚ÜíM4 gate items **#1** (native-
> dispatch / HTTP-bridge auth gap) and **#4** (`is_init_context` ‚Üí Admin UCAN).
> Blocked on ADRs [D-04-01](../../../../decisions/0015-ucan-capability-model.md)
> and [D-04-05](../../../../decisions/0016-native-dispatch-identity-threading.md)
> (both Accepted). Anchors verified against `main` @ `0d72d30`.
>
> Four ambiguities in `task.md`/the ADRs were owner-confirmed on 2026-07-14 ‚Äî see
> **Resolved Decisions** at the bottom. Headline: per-caller authorization of
> native services is ultimately **FDAE (M04B)**; B0 lays an interim
> identity-present gate and leaves `TODO(M04B/FDAE)` reminders at each enforcement
> point.

## 0. Summary of the change

1. New crate `syneroym-ucan` (`crates/ucan`) ‚Äî the **capability type model** only
   (D-04-01 ¬ß1,3): `ResourceUri`, `Ability`, `Capability`, `SessionContext`, plus
   pure ability-entailment/attenuation functions. `CapabilityToken`/`issue`/
   `verify_chain` are **deferred to B1** (see Flag #2).
2. `syneroym-rpc` gains `CallerContext` + `AuthLevel` (D-04-05 ¬ß1,2) and a
   `caller: CallerContext` field on `NativeInvocation`.
3. Router resolves a `CallerContext` from the (now-mandatory) preamble handshake
   and threads it into every native dispatch; anonymous callers to a
   native-capability interface / HTTP bridge are rejected **before** dispatch.
4. `creator_id` at write sites becomes the caller, not the callee service.
5. The two `is_init_context` / `TODO(M4)` gates (guest + native `execute-ddl`)
   are replaced by an Admin-capability check (`data-layer/admin`), with
   init/migrate running as `AuthLevel::LocalElevated`.
6. `SubstrateConfig` gains `[iam].admin_ucan_root` (interim Admin allowlist until
   B1 wires full UCAN verification).

The whole workspace recompiles as one ‚Äî `NativeInvocation` is an internal type,
no WIT surface changes, no compat shim (D-04-05 Consequences).

> **Added scope (beyond task.md) ‚Äî client-side identity, ¬ß0.5.** Making verify
> mandatory rejects *anonymous* (no-pubkey) callers. Every internal client today
> sends **no pubkey and no delegation**, so without ¬ß0.5 this slice breaks the
> entire workspace (roymctl, gateway, most integration/e2e tests). ¬ß0.5 gives the
> SDK/gateway a self-asserted identity so "anonymous" means a genuinely
> credential-less peer ‚Äî the exact peer task.md's failure test wants rejected.
> This is the concrete realization of the owner-confirmed Flag #1 latitude.

## 0.5 Client-side identity (prerequisite ‚Äî must land with ¬ß4, or the workspace regresses)

**Problem (reviewer-confirmed).** `RoutePreamble::binary_json_rpc`
(`crates/router/src/preamble.rs:294`) hard-codes `pubkey: None, delegation: None`,
and so does `passthrough_with_conn` (`crates/sdk/src/lib.rs:586-587`). The SDK
builds an ephemeral Iroh endpoint with no syneroym `Identity`
(`connect_with_mechanisms`, sdk/src/lib.rs:201 `Endpoint::empty_builder()`).
`verify_preamble` requires `pubkey` unconditionally (handshake.rs:47-50) ‚Üí every
SDK/gateway call resolves to the **anonymous** bucket. Once ¬ß4.3 rejects `None`
callers on native interfaces, these all break: `roymctl` security + orchestrator
ops, `crates/client_gateway/src/gateway.rs:172` HTTP-proxy passthrough, and the
`basic_lifecycle`/`podman_lifecycle`/`messaging_client_e2e`/`stream_client_e2e`/
`http_passthrough_e2e` + `coordinator_iroh` tests (deploy/`readyz`/`wait_for_ready`
poll the orchestrator interface; messaging/http_passthrough hit the 5 capability
interfaces directly). Conflicts with the "no M0‚ÄìM3C regression" exit criterion.

**Fix.** Give internal clients a *self-asserted* identity (pubkey only, no
delegation ‚Äî verify_preamble's no-delegation branch treats the source key as its
own master, handshake.rs:87-91). A verifiable pubkey is all B0's gate requires;
FDAE (M04B) decides whether that DID is actually authorized.

1. **`SyneroymClient`** (`crates/sdk/src/lib.rs:46`) gains an
   `identity: syneroym_identity::Identity` field (generate in `new`/`new_with_*`,
   or accept one ‚Äî see below). Add a stable-identity constructor
   `SyneroymClient::new_with_identity(...)` so callers that need a *known* DID
   (roymctl, gateway) can pass one; `new` generates an ephemeral one.
2. **`RoutePreamble::binary_json_rpc`** (preamble.rs:294): either add a `pubkey`
   parameter, or (cleaner) leave it identity-less and have the SDK set
   `preamble.pubkey = Some(hex(self.identity.public_key().to_bytes()))` after
   construction in `open_request_stream` (sdk/src/lib.rs:334). Pick one; the
   SDK-sets-it variant avoids touching every `binary_json_rpc` caller.
3. **`passthrough_with_conn`** (sdk/src/lib.rs:568-587): set `pubkey` from the
   passed/held identity instead of `None`.
4. **`client_gateway`** (gateway.rs:153,172): construct its `SyneroymClient` via
   `new_with_identity` using the **node's own identity** (`identity.key`, the same
   key `setup_substrate_identity` loads ‚Äî the gateway runs in-process and already
   has access to it). Owner decision 2026-07-14: **node DID for B0**, not the
   substrate-owner (controller) DID ‚Äî see the owner-DID follow-up note below.
5. **`roymctl`** management commands: same ‚Äî construct the client with an
   identity. Because B0 does **not** hard-gate the control-plane security/
   orchestrator interface (¬ß8.1, softened), roymctl does *not* need any admin key;
   it only needs *an* identity so it isn't anonymous. It may generate/hold its own
   self-asserted CLI identity (a distinct caller DID is fine).

> `TODO(M04B/FDAE)` reminder at the SDK identity site: a self-asserted pubkey is
> an *assertion*, not proof-of-possession (the no-delegation handshake path does
> not challenge it). B1/M04B tighten this to verified UCAN chains; B0 only needs
> "not anonymous."

### 0.5.1 Owner-DID threading ‚Äî deferred (owner decision 2026-07-14)

The substrate **owner** (`controller_did` / `ControllerAgreement.controller`,
`crates/substrate/src/identity.rs`) is a *distinct* DID from the node identity.
Making the gateway present the **owner** DID as caller (so a remote substrate sees
*who* owns the calling node, not just the node) would need an owner‚Üínode
`DelegationCertificate` (`master = controller_did`, `temp = node key`) in the
preamble ‚Äî which the handshake already supports (`verify_preamble` resolves
`master_did` from the delegation), **but no such artifact is issued anywhere in
the repo today** (only the `ControllerAgreement`, a different serialization; every
`DelegationCertificate::issue` is a test/bench). Per owner decision, **B0 presents
the node DID only**; threading the owner DID via a provisioned owner‚Üínode
delegation is a **later slice / setup task**. Leave a reminder at the gateway
client-construction site:
```rust
// TODO(post-B0): present the substrate-owner (controller) DID as caller by
// carrying an owner->node DelegationCertificate here (verify_preamble already
// resolves master_did from it). Requires provisioning that owner-signed
// delegation (none exists yet ‚Äî only ControllerAgreement). B0 uses node DID.
```

// Always attempt; `None` means "no verifiable identity", tolerated only by

        if preamble.delegation.is_some() {

    -> CallerContext
{
    let now = /* unix secs */;

### 4.4 Enforcement summary + FDAE reminder (Flag #1 resolution)
Native services are reachable **only** through `dispatch_json_rpc_once`'s Native
arm and the `messaging/subscribe` arm ‚Äî both now reject a `None` (anonymous, no
pubkey) caller. Relay forwarding (io.rs:103-164), TCP proxy, and raw
stream-protocol passthrough never reach these, so health/relay/coordinator
traffic without a pubkey is unaffected.

**B0 decision (per owner):** a *self-asserted* caller (valid pubkey, no
delegation) is **allowed** through the gate and threaded as `creator_id`. B0 only
enforces "some verified identity is present." The real per-caller authorization ‚Äî
that only the **service owner** or **substrate owner** may call a given native
service, and finer row/column limits ‚Äî is **FDAE config (M04B)**, not B0. Leave a
reminder at the enforcement point in `dispatch.rs` (Native arm) and at
`build_caller` in `io.rs`:
```rust
// TODO(M04B/FDAE): B0 gate only proves *an* identity is present. Which callers
// may actually reach this native service (service-owner / substrate-owner) and
// with what row/column scope is enforced by the FDAE policy engine (M04B),
// evaluated against `caller.session`. Until then any verified identity passes.
```

### 5.3 Reject anonymous bridged capability requests ‚Äî enforcement location (fixes reviewer #4)
Rejection is inherited from ¬ß4.3 (the Native arm returns an error for `None`),
surfaced as HTTP 500. To return a cleaner **401** *without* gating the
`stream`/chunked-upload route (which ¬ß5.5 keeps open) or the self-authorizing blob
GET (¬ß5.4), do **not** put the check at the top of `dispatch_route` (http.rs:526)
‚Äî that fans out to all three targets incl. `stream`. Instead:
- Add the `self.caller.is_none() ‚Üí 401` guard inside the shared `dispatch_native`
  free fn (http.rs:146) **only** ‚Äî every data-layer/messaging bridged route and
  the JSON-RPC fallthrough reach native dispatch through it, so one guard covers
  them all and maps to 401.
- Blob GET's internal dispatches (¬ß5.4) call `dispatch_native` with an explicit
  `service_system` caller (never `None`), so they are never 401'd.
- The `stream` route (¬ß5.5) never calls `dispatch_native`, so it is untouched.

### 5.4 Signed-URL blob GET is self-authorizing (Flag #3) ‚Äî must bypass `self.dispatch()` (fixes reviewer #3)
`handle_blob_get` (http.rs:826) authorizes via `crypto::verify_signed_url` (HMAC),
independent of the connection's delegation. Its internal dispatches must run with a
`service_system` caller, **not** `self.caller` (which may be `None` for a
signed-URL request on a delegation-less connection). **Critical:** the current
`handle_blob_get` `open-download` call is written `self.dispatch("blob-store",
"open-download", ‚Ä¶)` (http.rs:875) ‚Äî but ¬ß5.2 pins `self.dispatch()` ‚Üí `self.caller`.
Following ¬ß5.2 literally would 401 an anonymous signed-URL GET, breaking the exact
case Flag #3 protects. So ¬ß5.4 **overrides** ¬ß5.2 for this path:
- Rewrite the `open-download` call to invoke the **free** `dispatch_native(...)` fn
  directly with `CallerContext::service_system(&self.preamble.service_id)` ‚Äî the
  same pattern `blob_download_step`/`BlobDownloadState::Drop` already use for
  `read-chunk`/`close-download` (http.rs blob_download_step, Drop). Those two also
  switch from `None`-less bare calls to passing the `service_system` caller.
- Store the `service_system` `CallerContext` in `BlobDownloadState` so
  `blob_download_step` and `Drop` reuse it.
Do **not** gate the blob GET on the connection's delegation.

**B0 decision (per owner, mirrors Flag #1):** the HMAC signature *is* the B0
authorization for this path; the `service_system` caller is the interim gate.
Final policy (who may fetch which blob) is FDAE's (M04B). Leave a reminder at
`handle_blob_get`:
```rust
// TODO(M04B/FDAE): the signed-URL HMAC is the B0 authorization for blob GET.
// Final policy (who may fetch which blob) is enforced by FDAE (M04B) against the
// resolved caller; `service_system` is an interim system identity.
```

### 5.5 Stream upload route (http.rs:741)

## 9.5 Cross-node proxy-hop identity seam (task.md B0 line; ADR-0016 ¬ß6) ‚Äî define, don't implement

task.md's B0 paragraph explicitly assigns "**Define how identity threads across a
cross-node proxy hop (A1 seam)**" to B0 (not A1). A1 depends on B0's
`NativeInvocation` shape *specifically so it does not re-thread identity* ‚Äî so B0
must state the seam now, even though A1 implements the proxy. **This was missing
from the plan (reviewer #2); it is design-only, no code in B0.**

**The seam (per ADR-0016 ¬ß6, and D-04-01 ¬ß3 "never deserialized-and-trusted from
the wire"):**
1. `CallerContext` / `SessionContext` are **always locally constructed** from a
   verified handshake (¬ß4.2 `build_caller`) or a substrate-injected lifecycle
   token (`local_elevated`). They are **never** serialized into, nor deserialized
   out of, the wire. `NativeInvocation.caller` therefore holds only
   locally-verified identity ‚Äî confirmed by the fact that every construction site
   in this plan builds it host-side.
2. A **cross-node proxy call (A1)** carries, in its *request envelope metadata*
   (not `params`, not `NativeInvocation`), the caller's DID **and its signed
   proofs** ‚Äî the `DelegationCertificate` today, UCAN `CapabilityToken`s once B1
   lands. Capabilities are never sent as a trusted list.
3. The **data-owning (destination) node** verifies those proofs locally (reusing
   ¬ß4.2's `build_caller` / B1's `verify_chain`) and constructs a **fresh**
   `CallerContext` before building the `NativeInvocation`. Enforcement always
   happens at the node that owns the resource.

**Consequence for A1 (why the shape is stable):** A1 adds a *wire envelope* type
and a verify-then-`build_caller` step at the destination; it does **not** modify
`NativeInvocation`/`CallerContext`. B0's type shape is final for this purpose.
Record this explicitly so A1 does not attempt to put `CallerContext` on the wire.
No B0 code beyond a doc-comment on `CallerContext` stating "locally-constructed;
never wire-serialized ‚Äî cross-node hops send signed proofs, re-verified at the
destination (ADR-0016 ¬ß6)."
