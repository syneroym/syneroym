## Critical — Security

- [ ] **1. Publish-side namespace spoofing: any caller can impersonate any other service's topic namespace.** `namespace_topic()` (`crates/mqtt_broker/src/lib.rs:67-69`) treats any topic starting with `svc/` as "already fully-qualified, use literally" — a rule `task.md` (lines 219-225) specifies only as a **subscribe**-side cross-service opt-in. Both publish call sites reuse it unconditionally:
  - `crates/sandbox_wasm/src/engine.rs:193-197` (`host_api::Host::publish`, WASM guest path) — `namespace_topic(&self.component_id, &topic)`.
  - `crates/control_plane/src/synsvc_native.rs:532` (`dispatch_messaging`, native/non-WASM path) — same call shape with the caller's own `service_id`.
  A malicious or buggy component/native caller publishes to `topic = "svc/<victim>/orders/new"`; `namespace_topic` returns it unchanged (the `starts_with("svc/")` branch short-circuits), so the message lands verbatim in `svc/<victim>/...` — indistinguishable from a message `<victim>` published itself. This is exactly the `task.md:854` Failure/Security Test row ("Service A publishes to service B's MQTT namespace → Delivery blocked") and it is **not** blocked.
  **Fix:** add a publish-specific namespacing function that unconditionally prefixes `svc/<caller_service_id>/` regardless of the topic's existing prefix (or explicitly rejects/escapes a caller-supplied leading `svc/`), and use it at both call sites; keep `namespace_topic`'s current literal-passthrough behavior for subscribe only.
  **Test gap:** `crates/sandbox_wasm/tests/messaging_integration.rs:113-134` only tests the subscribe opt-in direction; add a test asserting a service subscribed to its own bare topic does **not** receive a publish spoofed via `svc/<victim>/...` from another service.

## High — Correctness & Concurrency

- [ ] **2. Router-level reader task leaks when the broker side closes first.** `crates/router/src/route_handler/dispatch.rs`, `handle_messaging_subscribe` (~lines 274-334): the `tokio::spawn` at line 302 (reading frames to detect client-initiated close) discards its `JoinHandle`. The outer `select!` loop (line 313-330) can also exit via `receiver.recv()` returning `None` (broker/engine-side close, line 316) — in that path the function returns and drops `writer`/`handle`, but the spawned reader task is never aborted and stays parked on `framing::read_frame` indefinitely.
  **Trigger:** substrate shutdown, or the `MqttBroker`'s internal forwarding task erroring out, while a native `SyneroymClient` still has an open `messaging/subscribe` stream.
  **Fix:** capture the `JoinHandle` from the `tokio::spawn` and `.abort()` it before returning from the broker-closed branch (mirror what `SubscriptionHandle::drop` already does for the broker-side forwarding task).

- [ ] **3. `SendStream::stopped()` dead-subscriber detection specified in `task.md` is not implemented.** `task.md` lines 358-365 ("Dead-subscriber cleanup") explicitly requires racing the `mpsc::Receiver` against `SendStream::stopped()` on the QUIC send half. In the actual code, `writer` in `handle_messaging_subscribe` is generic (`W: AsyncWrite + Unpin + Send + 'static`, `crates/router/src/route_handler/dispatch.rs:277,284`) — it is never the concrete Iroh `SendStream` type, so `.stopped()` cannot be and is not called anywhere in this file. The only close-detection path is reading the **read** half until EOF/error (lines 301-311), which detects the client finishing its own send side, not a peer that issues `STOP_SENDING` on the receive direction without otherwise closing.
  **Trigger:** a peer that stops wanting the subscription pushed to it (QUIC-level `STOP_SENDING`) without closing its connection or its own send half. The subscription is only discovered dead on the next failed `write_frame` (line 324) — or never, for a low-traffic topic.
  **Fix:** thread the concrete `SendStream` (not a generic `W`) into this path so `.stopped()` can be raced in the `select!` alongside `receiver.recv()` and the reader-EOF signal, per the design note.

- [ ] **4. `subscribe()` persists the DB row before the live registration can fail, with no rollback.** `crates/sandbox_wasm/src/engine.rs:199-219`: `save_messaging_subscription` (line 206) runs and commits *before* `engine.upgrade()` is checked (line 210). If `upgrade()` returns `None` (engine mid-teardown), the function returns `Err`, but the `messaging_subscriptions` row is already persisted with no corresponding live broker registration.
  **Trigger:** a guest's `subscribe` call races `AppSandboxEngine` shutdown/hot-reload.
  **Fix:** either check `engine.upgrade()` before the DB write, or delete the row on the `upgrade()`-failure path before returning `Err`.

- [ ] **5. `unsubscribe()` reports success while leaving the live subscription active.** `crates/sandbox_wasm/src/engine.rs:221-236`: `delete_messaging_subscription` (line 228) runs unconditionally, but the live in-memory removal only happens `if let Some(engine) = engine.upgrade()` (line 232) — no `else` branch, and the function returns `Ok(())` regardless (line 235).
  **Trigger:** same teardown race as #4. Caller is told unsubscribe succeeded; the broker keeps delivering; the DB row is already gone so `undeploy`'s DB-driven cleanup and restart-replay can't rediscover it either.
  **Fix:** return an error (or otherwise surface the inconsistency) when `upgrade()` fails here instead of silently returning `Ok(())`.

- [ ] **6. Per-subscriber forwarding task blocks instead of dropping messages past the bound, contradicting the documented spec.** `crates/mqtt_broker/src/lib.rs:171`: `sender.send((topic, payload)).await` is a blocking bounded-channel send. `task.md` lines 366-372 explicitly specifies: "a subscriber that can't keep up gets messages dropped past the bound rather than blocking the broker's delivery loop" (native subscribers), and the WIT-facing backpressure story generally treats this path as best-effort/at-most-once. Using `.send().await` instead of `try_send` means a slow consumer instead stalls this task's draining of `link_rx.next()`, which back-pressures into rumqttd's own per-link buffer rather than dropping at the documented boundary.
  **Fix:** use `sender.try_send(...)`, dropping the message (and optionally logging) on `TrySendError::Full`, matching the publish-side `try_publish` pattern already used elsewhere in this file.

- [ ] **7. `SubscriptionHandle::drop` calls a blocking, synchronous `rumqttd` call directly inside `Drop`.** `crates/mqtt_broker/src/lib.rs:216-226`: `link_tx.unsubscribe(...)` (line 223) is confirmed (via `rumqttd` 0.20.0 docs) to be a **synchronous, blocking** method — `rumqttd::local::LinkTx` also exposes a non-blocking `try_unsubscribe()` that isn't used here, mirroring the `publish`/`try_publish` split this same file already gets right elsewhere (`crates/mqtt_broker/src/lib.rs:119` uses `try_publish`). Calling the blocking variant from a synchronous `Drop` impl that can run on a Tokio worker thread risks stalling that worker (and every task multiplexed onto it) if rumqttd's internal event channel is momentarily full.
  **Fix:** switch to `try_unsubscribe()` in `Drop`, matching the non-blocking discipline already used for publish.

- [ ] **8. Duplicate `subscribe()` for the same topic can cause a brief double-delivery window.** `crates/sandbox_wasm/src/engine.rs:199-219` / `register_internal_subscription`: the DB write is idempotent (`ON CONFLICT DO NOTHING`), but the broker-level registration is not guarded — a second `subscribe()` call for a topic already subscribed opens a *second* `rumqttd` link before the first `SubscriptionHandle` is replaced/dropped from the in-memory map.
  **Trigger:** a guest calls `subscribe(topic)` twice for the same topic (e.g. retry after a transient error it doesn't distinguish from "already subscribed"). In the window between the second link subscribing and the first handle's `Drop` running (which unsubscribes link 1), a publish landing on that topic is forwarded to both links, and `handle-message` is invoked twice for one logical message.
  **Fix:** make `register_internal_subscription` check for an existing live handle for `(service_id, topic)` before opening a new broker link, returning success without re-subscribing if one exists.

- [ ] **9. `publish()`'s error mapping discards the real cause, always reporting "backpressure".** `crates/mqtt_broker/src/lib.rs:119-121`: `try_publish(...).map_err(|_| MessagingError::Internal("broker channel full: backpressure".to_string()))` collapses every possible `LinkError` (including e.g. a genuinely dead/closed router link) into the same backpressure message.
  **Trigger:** any non-backpressure `LinkError` (router thread died, link closed) is reported to callers/logs as transient backpressure, misleading anyone diagnosing a real failure or retry logic that treats "backpressure" as retryable.
  **Fix:** include `{e}` in the `Internal(...)` message, or match on the error kind and only use the "backpressure" wording when it's actually a full-channel condition.

- [ ] **10. `channel_capacity = 0` in config panics on first subscribe instead of surfacing a config error.** `crates/core/src/config.rs:745` (`MessagingConfig.channel_capacity: u64`, no lower-bound validation) flows through `crates/router/src/route_handler.rs:162` (`config.mqtt.channel_capacity as usize`) into `crates/mqtt_broker/src/lib.rs:159` (`mpsc::channel(self.channel_capacity)`), and `tokio::sync::mpsc::channel(0)` panics (`assert!(buffer > 0)`).
  **Trigger:** an operator sets `[mqtt] channel_capacity = 0` in the substrate TOML. Substrate panics on the first subscribe (including the startup replay of any persisted `messaging_subscriptions` row), rather than failing with a clear config-validation error.
  **Fix:** validate `channel_capacity > 0` when loading `MessagingConfig` (or in `MqttBroker::new`), returning a structured config error instead of panicking downstream.

## Medium — Architecture / Efficiency

- [ ] **11. All publish traffic substrate-wide serializes through one `Mutex<(LinkTx, LinkRx)>`.** `crates/mqtt_broker/src/lib.rs:117-118` (`self.host_link.lock().await`) is a single shared lock used by every WASM guest publish and every native `dispatch_messaging` publish across the entire process. `try_publish` itself is fast/non-blocking, so this is a contention point rather than a correctness bug, but it caps publish throughput on total substrate-wide volume rather than per-service/per-topic volume. Worth a comment noting this is a deliberate accepted tradeoff, or moving to a per-caller link if publish volume becomes a bottleneck.

- [ ] **12. The native `messaging/subscribe` special-case is a narrow, hardcoded bandaid, not a generalizable long-lived-stream mechanism.** `crates/router/src/route_handler/dispatch.rs` special-cases `interface == "messaging" && method == "subscribe"` as a string match on the very first frame, ahead of the generic `dispatch_json_rpc_once` path. Slice 6B (`task.md` lines 472-663: `register-stream-protocol`, `stream-cursor`, `stream-sink`) will need essentially the same "hold the writer open, special-case before generic dispatch" shape for different interfaces/methods. As written, that likely means copy-pasting another hardcoded interface/method check rather than reusing a declared mechanism (e.g. a small registry/trait marking which `(interface, method)` pairs require a long-lived stream). Worth generalizing now, before Slice 6B compounds the pattern.

- [ ] **13. Namespace isolation is a call-site convention, not a structural guarantee.** `MqttBroker::publish`/`subscribe` (`crates/mqtt_broker/src/lib.rs:117,148`) take a raw `&str` topic that callers must have already namespaced correctly — there's no type-level distinction between a raw guest-supplied topic and a namespaced one. Correctness currently depends on all 3 call sites (`engine.rs`, `synsvc_native.rs`, `dispatch.rs`) remembering to call the (correct) namespacing function. `task.md`'s own Slice 7 section says the SSE bridge will call "the same broker API Slice 6A's native push-delivery path uses, reused directly" — i.e. a 4th call site is already planned. Consider a `NamespacedTopic` newtype that can only be constructed via the namespacing function(s), so a future call site that forgets to namespace fails to compile rather than silently breaking isolation.

## Medium — Test Coverage

- [ ] **14. Performance budgets are measured but never asserted.** `crates/substrate/tests/messaging_client_e2e.rs:191-194` and `crates/sandbox_wasm/tests/messaging_integration.rs:184-187` both compute p99 latency and `eprintln!` it, with no `assert!` against the 5ms/25ms budgets `task.md`'s "Performance Budgets" table and "Measurable Exit Criteria" require. A regression that blows through either budget will not fail CI.
  **Fix:** add `assert!(p99 < Duration::from_millis(5))` / `25` (with headroom as appropriate for CI variance).

- [ ] **15. Only the `+` wildcard is unit-tested; `#` never is.** `crates/mqtt_broker/src/tests.rs:23-35` tests `sensors/+/temp` only. `task.md`'s day-1 spike and Acceptance Criteria (line 457) call for both `+` and `#` passthrough. Add a `#` multi-level wildcard test, especially given the `svc/` namespace-prefixing logic appends a segment ahead of the filter — worth confirming that composition doesn't produce an invalid filter when the caller's topic itself ends in `#`.

- [ ] **16. Retained-message support has no reachable production caller.** `crates/mqtt_broker/src/tests.rs:37-55` only exercises `publish_retained_for_test`, a `#[cfg(test)]`-only helper (`crates/mqtt_broker/src/lib.rs:129-143`). The production `publish()` path (`try_publish`, line 119) never sets the retain flag, and the WIT surface has no retain parameter by design (per `task.md`'s "A4" decision). This means "Wildcard topics and retained messages function correctly" (Acceptance Criteria) is unverifiable through any real API today — the test proves the mechanism works, not that any caller can invoke it. Worth a one-line status.md note that retained-message support is broker-internal/plumbing-only in Slice 6A, with no guest/native-facing trigger yet (consistent with the "A4" decision not to add a `retain` parameter now).

- [ ] **17. Restart-replay test bypasses the real startup path.** The restart-replay test in `crates/control_plane/src/service.rs` (`test_messaging_subscriptions_survive_restart_replay`) hand-rolls its own replay loop calling `register_internal_subscription` directly, rather than exercising the actual production replay wiring (wherever `RouteHandler::init` / substrate startup does the replay described in `task.md` lines 250-255). A regression that breaks, removes, or reorders the *real* startup replay (e.g. moving it after the router starts accepting connections) would not be caught by this test.
  **Fix:** add (or convert this test to) one that goes through the real startup entry point, or add a second test that does, alongside keeping the current unit-level one.

- [ ] **18. Native `messaging/publish` round-trip test doesn't prove delivery.** `crates/control_plane/src/service.rs` (the extended `test_native_dispatch_data_layer_and_blob_store_round_trip`, messaging block ~line 1390-1400) only asserts `publish_resp.payload == Value::Null` — i.e., that the RPC call didn't error, not that the message was actually published to the broker in a form a subscriber would receive.
  **Fix:** attach a subscriber (native or via `MqttBroker::subscribe` directly) to the same topic before the native publish call and assert it receives the payload.

- [ ] **19. Two brittle/flaky tests depend on timing or shared machine state.**
  - `crates/mqtt_broker/src/tests.rs:83-99` (`publish_returns_backpressure_error_when_channel_saturated`) relies on 20,000 fire-and-forget `try_publish` calls outracing rumqttd's router-drain thread; on a fast/idle machine the router may keep up and the loop finishes without ever observing backpressure, failing the `assert!(saw_backpressure)` even though nothing is wrong.
  - `crates/mqtt_broker/src/tests.rs:106-115` (`no_network_listener_is_bound`) binds the real, well-known port `127.0.0.1:1883`; it fails on any dev machine or CI runner that already has a local MQTT broker (e.g. mosquitto) or anything else on that port, unrelated to `MqttBroker`'s actual behavior.
  Both are testing real properties worth keeping, but should be made deterministic/hermetic (e.g. a configurable/injectable "simulate full" hook for the first; bind an ephemeral port and compare against a broker-exposed "no listener" API, or skip cleanly if 1883 is already taken, for the second).

- [ ] **20. E2E close-as-unsubscribe assertion has a race window.** `crates/substrate/tests/messaging_client_e2e.rs` (~line 200-210): after the client stream is closed, the test immediately publishes and asserts the client receives nothing further. If that publish reaches the broker before the router-side unsubscribe (FIN propagation → reader-task detection → `SubscriptionHandle` drop) completes, the message can still be delivered and the assertion flakes — unrelated to a real regression. Consider polling/retrying the post-close publish with a short delay bound, or asserting eventual (not immediate) cessation.

## Low — Reuse / Simplification / Efficiency / Altitude / Conventions

- [ ] **21.** `crates/router/src/route_handler/dispatch.rs:254` — the new `handle_binary_stream`'s dispatch-then-write-frame block duplicates `handle_json_rpc_loop`'s body (lines ~193-202) verbatim instead of sharing a helper. Extract a common `dispatch_and_write_frame(...)` used by both.

- [ ] **31.** `crates/core/src/test_constants.rs:33` — the new `messaging_pubsub_test_wasm_path()` writes `std::path::PathBuf::from(...)` despite `use std::path::PathBuf;` already being imported at line 4 (AGENTS.md "Mandatory Import Cleanup": *"Types via standard `use`... proactively remove inline fully-qualified paths"*). ## Informational (no action needed)

- **Migration approach is fine as-is.** `run_m3b_migration` (`crates/data_db/src/sqlite.rs:479-`) uses `CREATE TABLE IF NOT EXISTS`, which is idempotent and appropriate for this pre-release dev phase. Do not add migration-versioning/framework machinery for this slice — an illustrative migration is sufficient until the project actually ships.
- **The "no leak" broker-drop test is honestly scoped, just narrowly named.** `crates/mqtt_broker/src/tests.rs:63-76` and the module doc (`crates/mqtt_broker/src/lib.rs:1-20`) already disclose that `rumqttd` 0.20.0 exposes no way to stop its internal router OS thread, and that this is an accepted, harmless leak until process exit. The test verifies what's actually true (forwarding tasks + receiver closure), not literal OS-thread termination. No bug here — just note in `status.md` that "Broker task terminates cleanly on substrate shutdown" (Acceptance Criteria) means the crate's own Tokio-side state, not the underlying OS thread, to avoid future confusion.
- **`rumqttd::Broker::new()` starting the router thread automatically** — see Reconciliation table row 1. Confirmed correct by design; no code change needed.
